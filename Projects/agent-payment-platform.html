Agent Payment Platform
Designing and specifying a multi-tenant agent portal for international cross-border payments — enabling counsellors at the world's largest cross-border placement agency to generate secure payment links for payers across dozens of countries and hundreds of universities.
Executive Summary

This project defined the complete product requirements for a new Agent Payment Platform serving the world's largest international cross-border placement agency. The platform enables counsellors (agents) to generate secure payment links for payers, supporting tuition, accommodation, living expenses, and visa fees across all onboarded universities. Built on a multi-tenant architecture, the platform supports a four-level organisational hierarchy (Platform, Agency, Master User, Sub-Accounts) and is designed for future expansion to additional agencies.

Key outcome: A comprehensive PRD and BRD were produced covering two payment flows (agent-initiated ~90%, payer-initiated ~10%), role-based permissions, product-to-beneficiary mapping, dashboard and reporting requirements, email notifications, and integration with existing Web and Operations infrastructure. The project advanced through kick-off meetings, design sessions, and stakeholder alignment with the agency partner.

Platform Architecture

Project Scope

In Scope (Phase 1)

  • Agent portal with authentication and 2FA
  • Payment link generation (up to 5 product categories)
  • Individual Payment Form with T&C acceptance
  • Dual payment flows (agent-initiated + payer-initiated)
  • Role-based permissions (4 levels)
  • Web DB and Operations integration
  • Product-to-beneficiary bank mapping
  • Email notifications (confirmation, reminder, receipt)

Out of Scope / Future Phases

  • Automated refund processing
  • Agency Internal System-to-Internal System API integration
  • Native mobile application
  • Agency self-registration
  • Commission visibility in agent portal
  • Advanced dashboard and reporting
  • University self-onboarding
Key Design Decisions

Critical architectural and business decisions shaping the platform

1
Multi-Tenant Architecture
Platform designed as agency-agnostic from day one. The hierarchy (Platform → Agency → Master User → Agents) supports onboarding additional agencies with full data isolation, avoiding agency-specific implementation.
2
Existing Infrastructure Reuse
Rather than building standalone systems, the platform integrates with existing Web DB and Operations pipelines — reusing transaction processing, product mapping, and file routing infrastructure already proven in production.
3
Payment Link Model
Agents never handle payer card details. The link-based payment model ensures payers always complete the payment step directly, maintaining PCI compliance and reducing fraud risk across all payment channels.

Lessons & Insights
What Worked Well

Comprehensive PRD Before Development

Producing a detailed product requirements document covering the full platform vision before beginning development ensured all stakeholders were aligned on scope, hierarchy, and phased delivery before any code was written.

What Worked Well

Agency-Agnostic Design Principle

Building the multi-tenant hierarchy as a platform feature rather than an agency-specific implementation means additional agencies can be onboarded in the future without structural changes, maximising the return on development investment.

Challenge

Open Questions Requiring Partner Input

Several critical decisions (data fields, link expiration, document collection) required input from the agency partner. Tracking these systematically and addressing them in recurring meetings prevented them from blocking progress.

Key Learning

Integration over Greenfield

Choosing to build on existing Web DB and Operations infrastructure rather than creating standalone systems significantly reduced scope, risk, and delivery time while maintaining consistency with existing transaction processing.
