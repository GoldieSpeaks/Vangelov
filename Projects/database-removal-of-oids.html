Database OID Removal & Modernisation
Systematically identifying, analysing, and replacing legacy Object Identifiers (OIDs) across the entire database layer — unblocking critical PostgreSQL version upgrades.
Executive Summary

The platform's PostgreSQL database contained legacy Object Identifiers (OIDs) — a deprecated feature that newer PostgreSQL versions no longer support. These OIDs were embedded across database schemas, application queries, models, and API endpoints, creating a direct blocker for database version upgrades. Without removal, the infrastructure team could not modernise the database engine, leaving the platform on an increasingly outdated version with growing security and performance risks.

Key outcome: A comprehensive technical analysis mapped every OID usage across the entire system. All OIDs were systematically replaced with standard identifiers (IDs) across the database schema, application logic, and API layer. Development was conducted on an isolated environment to prevent interference with parallel projects, then merged to the main development environment for staged testing. The project was released as a dedicated deployment, confirming zero regression across all affected systems.

Lessons & Insights
What Went Well

Isolated Development Environment

Using a separate development environment for the OID rework was critical. The changes touched a large number of interrelated files and systems, and isolating them prevented any disruption to other active projects during the multi-week development period.

Learning

Analysis-First Approach Pays Off

Investing in a thorough technical analysis before any code changes reduced rework risk. The comprehensive audit document served as both a development guide and a testing checklist, ensuring no OID references were missed during replacement.

Consideration

Legacy Debt Accumulation

OIDs had been embedded in the system for years without causing visible issues — until they became an upgrade blocker. This project highlights the value of periodically reviewing technical debt before it becomes a constraint on infrastructure modernisation.

Risk Managed

Cross-System Regression Risk

Replacing identifiers across every layer of the application stack carried significant regression risk. The staged testing approach — isolated environment, then main dev, then internal QA, then external QA — provided multiple safety nets that caught issues before production.
