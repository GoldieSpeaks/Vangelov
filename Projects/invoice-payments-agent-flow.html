Case Study

B2B Invoice Payment API

Extending an cross-border payments platform to support business-to-business invoice payments via API — enabling third-party agencies to pay institutions directly using invoice numbers, with custom margin controls and multi-system integration.

~3 Months End-to-End Delivery
Cross-System API + Web + Operations
5 BRD Versions Requirements Maturity
B2B First New Payment Paradigm

E Executive Summary

This project introduced a fundamentally new payment model to the cross-border payments platform: B2B invoice payments via API. Previously, the API supported only individual consumer-to-business (C2B) payments requiring payer data. The new flow enables third-party agencies to register payments using only an invoice number and corporate details — no payer information required. The development spanned API endpoint creation, database schema extensions, operations application updates, FX conversion logic changes, compliance screening integration, and configurable per-agency margin controls.


Key outcome: The platform now supports both individual and corporate payment flows through a unified API, with agencies able to transact on behalf of institutions using bank transfers only, custom margins, and a dedicated operations management interface. Multiple release version adjustments reflected the complexity and cross-team coordination required.

💡 Lessons & Insights

What Worked Well

Thorough BRD Process

Five BRD iterations ensured all edge cases, compliance requirements, and cross-system impacts were identified before development began. This eliminated major rework during the build phase.

What Worked Well

Dual Team Sign-off

Requiring approval from both Web and Operations leads ensured alignment between the two development streams and prevented integration issues at deployment time.

Challenge

Legacy Data Model Constraints

Field-length inconsistencies across systems surfaced during analysis. The team could not modify existing schema without risk, requiring a new validation layer as a workaround — adding complexity.

Key Learning

Scope Discipline with Refund Deferral

Acknowledging that refund functionality was needed but deliberately deferring it kept the initial release focused. This prevented scope creep in an already complex multi-system project.

Risk Identified

Multi-Release Coordination

The project required changes across four release versions before final deployment, reflecting shifting priorities and dependencies. Tighter upfront release planning could have reduced rescheduling overhead.

Key Learning

C2B to B2B Paradigm Shift

The API was originally designed around individual individual payments. Retrofitting it for corporate B2B flows required rethinking data models, validation, and compliance — a useful lesson for future API extensibility planning.
