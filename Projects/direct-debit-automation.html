Direct Debit Automation
Replacing a fully manual direct debit refund process with automated indicators, beneficiary storage, and multi-currency reporting — eliminating error-prone email-based workflows.
Executive Summary

The direct debit refund process for API-integrated clients was entirely manual: the payments team relied on email requests to determine whether refunds should be collected via direct debit or bank transfer. There was no system indicator, no stored DD details, and no reporting capability. This project automated all three gaps.

Key outcome: The back-office system now automatically determines refund payment methods based on beneficiary configuration, stores direct debit account details with role-based access controls, and generates consolidated multi-currency DD reports. The project was deployed to production and validated in the live environment since the functionality was new and had no existing production data to test against internally.

Development Breakdown
Refund Method Indicator
Database-driven logic: when a beneficiary is flagged as direct debit (is_directdebit=1), refunds automatically default to DD. Otherwise, bank transfer. This replaced the email-based manual determination entirely, with the trigger executing on refund line creation.
DD Account Storage
New capability to store direct debit account details directly on beneficiary records within the back-office system. Only super users and payment team members can add or edit DD entries, enforced through permission checks and error messages for unauthorized attempts.
Consolidated Reporting
New report type under agent reports: a consolidated CSV export of all DD refund lines within a date range, covering four currencies (EUR, CAD, GBP, USD). Includes DD account details from which funds will be collected. Fixed to show only the selected DD account per line.

Lessons & Insights
What Went Well

Pragmatic Production-First Testing

The decision to test in production was not avoidance of process but a pragmatic choice: the DD functionality had no existing data in test environments. Validating with real data ensured meaningful results rather than synthetic test scenarios.

Learning

Report Requirements Need Specificity

The GBP filtering issue revealed that the BRD specified a consolidated report but did not clearly define per-currency filtering. QA caught the gap, leading to a clarification that a single consolidated report was the intended behavior.

Consideration

Security Controls as Day-One Features

The permission bypass for DD editing was caught during testing. For financial data like direct debit details, role-based access controls should be part of the initial specification, not discovered during QA.

Risk Noted

Blank Payment Method Validation

The ability to save a blank refund payment method was identified as a defect. While not blocking release, it represents a data integrity risk that requires resolution to prevent downstream processing errors.
