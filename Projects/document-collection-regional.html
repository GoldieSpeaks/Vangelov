Executive Summary

A regional payment processing partner required that payers from specific countries upload regulatory documents before their transactions could be processed. Previously, document collection relied on email-based instructions — an approach that frequently resulted in missed submissions, expired transactions, and delayed fund matching.

This project delivered a fully integrated document upload workflow embedded directly into the payment process on Web Payment Platforms. The solution enforced mandatory document submission by country and payer type, introduced time-bound transaction expiration, and automatically packaged and transferred documents to the partner via a secure regional file transfer server compliant with data residency regulations. The implementation spanned multiple platform templates, database schema changes, and cross-team coordination with infrastructure and the external payment partner.

Project Scope

In Scope

  • Mandatory document upload page for two regional corridors
  • Country-specific and payer-type-specific document requirements
  • Time-bound transaction expiration (24-hour window)
  • ZIP packaging and SFTP transfer to regional cloud server
  • Upload progress indicators and UI feedback
  • Payment instruction updates for both corridors
  • Database schema changes (4 new tables, column extensions)
  • Multi-template styling updates across platform variants

Out of Scope

  • Operations system automation (deferred to follow-up)
  • External QA test cases (partner-side testing)
  • Risk assessment documentation
  • Automated reminder emails for pending documents
  • Hyperlink popup (released as post-launch enhancement)

Key Decisions

Critical decisions shaped by regulatory requirements and partner coordination

When Decision Decision Maker Type
Week 4 Set transaction expiration to 24 hours. Initially prototyped at 72 hours for testing, then reduced per stakeholder agreement at sign-off call. Product Manager Strategic
Week 5 Separate upload logic into a new plugin. Core plugin refactored: transaction listing separated from document upload form generation for maintainability. Lead Developer Technical
Week 5 Store documents on regional cloud server. Data residency regulations required files from Corridor A to be stored within the originating country's borders. Product Manager Compliance
Week 6 Defer popup enhancement to post-release. Hyperlink information popup assigned to a different developer to avoid blocking the main release. Product Manager Process
Week 6 Partially uploaded documents: hide completed slots. If a user uploads some but not all documents, only remaining required fields are shown on return visit. Product Manager UX

Lessons & Insights

What Worked Well

Plugin Architecture Separation

Refactoring the upload system into separate plugins for transaction listing and document upload created a cleaner, more maintainable codebase. Each concern could be modified independently without risking regressions.

What Worked Well

Partner Prototype Validation

Sending UI prototypes and document field descriptions to the payment partner for approval before final development ensured alignment on regulatory requirements and prevented costly post-release corrections.

Challenge

Three Release Reschedules

The release was moved across three consecutive monthly windows. While each reschedule was justified by competing priorities, the cumulative delay extended the project timeline beyond initial expectations.

Key Learning

Decouple Enhancement from Core Release

The decision to defer the hyperlink popup to a separate developer and release cycle was effective. It prevented a single developer's availability from blocking the primary feature delivery.

Technical Debt Identified

Operations Automation Deferred

The automated transaction cancellation logic for the back-office system was deferred. Until implemented, transactions without documents are handled by the standard 14-day auto-cancellation rather than the 24-hour window.

Key Learning

Data Residency Drives Architecture

The requirement to store documents within specific geographic boundaries added infrastructure complexity (regional cloud server provisioning) that needed to be addressed before development could begin.
