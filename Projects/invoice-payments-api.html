B2B Invoice Payments API
Building a New Payment API Using Existing Systems as Reference

This project involved building a completely new payment API for B2B invoice payments, using the existing individual payment system as reference for architecture and business logic. The Invoice Payment API enables companies and agencies to pay universities and institutions via invoice numbers, creating a parallel system for agency-centric B2B invoice-based payments.

As Product Owner, I led the design and development of a new payment API architecture, creating data models, payment logic, FX conversion rules, and system integrations specifically for the B2B workflow. The project required building a parallel system that handles payer identification, margin calculations, payment method filtering, and CRM integration while referencing the existing individual payment flows for consistency.

The Challenge
New API Use Case
The existing payment services and infrastructure were built specifically for individual payments. Building a new B2B invoice payment system required creating entirely separate infrastructure while using the existing system as reference.

The core challenge was building a completely new API from scratch while using the existing individual payment system as architectural reference. The new B2B invoice-payment use case required creating separate data models and payment flows that operate independently from the individual system, while maintaining consistency in areas like FX conversion and CRM integration.

This required careful architectural decisions: designing new payment flows and interfaces with agency-specific parameters, implementing custom per-agency margin calculations, supporting multiple payment methods for invoice payments, creating new FX conversion logic, and extending the TM EDU CRM system to manage agencies as distinct entity types alongside the existing individual system.

Creating Separate Data Models for Agency Payments

The new API was designed specifically for agency information (company name, invoice number, corporate details). This created a clean separation from individual payment data models, allowing each system to operate independently while using the existing infrastructure as reference for consistency.

Payment Method Filtering and Processing Selection

The new API supports multiple payment methods suitable for B2B invoice transactions, with eligibility rules that ensure appropriate processing for agency payments.

Custom Per-Agency Margin Calculation

The new API needed to support flexible pricing models for different partner agencies. This required implementing a custom margin system that could apply differential rates based on agency relationships.

New FX Conversion Logic

The new API required implementing custom FX conversion logic to handle currency conversion for B2B invoice payments. This included automatic currency routing and settlement calculations specifically designed for agency-to-university transactions.

CRM System Integration for Agency Management

The TM EDU CRM was built to manage individuals and institutions. We needed to extend it to manage agencies as distinct entity types, creating new windows for API Invoice Payers with corporate information, transaction history, and beneficiary relationships‚Äîmatching the functionality of individual windows.

Reference Number Generation and Data Differentiation

API invoice payments needed to be clearly differentiated from individual payments throughout all systems. We introduced new reference numbering logic, data structures, and CRM records that distinguished API invoice transactions from traditional individual payments.

Key Implementation Details
How It Works
Dual-Path API Architecture

The new API provides dedicated capabilities for invoice payments with agency-specific parameters. The system processes invoice payments separately from individual payments, using different data models and validation rules while referencing existing patterns for consistency.

Agency Unique Identification

Agencies are identified by the combination of (Web Login ID + Payer Unique ID), which ensures uniqueness across partners. The CRM system checks if an agency already exists using this criteria and either creates a new agency file or reuses the existing one, preventing duplicate records.

Payment Method Selection Logic

When an agency requests FX rates through the new API, the system returns available payment methods based on eligibility rules for invoice transactions.

Margin Calculation Engine

The margin system supports custom pricing for specific agency partnerships with fallback to default rates, enabling partners to negotiate different pricing structures based on business relationships.

FX Conversion Logic

The new API implements automated currency conversion logic with fallback mechanisms to ensure transactions can be processed regardless of geographic coverage limitations.

CRM Integration Points

Extended the TM EDU CRM APP with a new "API Invoice Payers" window specifically for managing agencies. This creates a separate management interface for corporate payers alongside the existing individual system.

Reference Number Generation

Invoice payments use unique reference numbering that clearly identifies them as B2B transactions, helping with reconciliation and tracking across systems.

Beneficiary Pre-Creation Process

Universities must be pre-configured with complete institutional and banking information before accepting invoice payments. The new API validates this data independently from the individual payment system.

Risk Management & Compliance
Risk Management
False Company Details Risk

Agency verification processes and compliance checks are implemented to ensure data accuracy and prevent fraudulent activity, with partners responsible for initial verification and ongoing monitoring.

Beneficiary Onboarding Requirements

Geographic coverage limitations are managed through clear documentation of supported regions and ongoing expansion of banking partnerships to increase global reach.

Reference Number Length Validation

The API validates reference numbers to ensure compatibility with banking systems, providing clear error messages and guidance when adjustments are needed.

Compliance Advantage Requirements

Both institutional customers and agency partners require proper compliance onboarding with automated workflows and documentation processes to ensure regulatory compliance.

Key Learnings
Key Learnings
Dual-Path Architecture Requires Clear Design

Building parallel systems for individual and agency payments required careful design to maintain consistency. The key was creating clear separation at the API level while using existing patterns as reference, ensuring partners understood the distinct interfaces for each payment type.

Data Model Changes Impact Downstream Systems

Creating new data models for invoice payments required coordinated development across the API, WEB DB, and CRM system. Each system needed to handle agency data separately from individual data. Planning the data architecture upfront and building integration points systematically ensured clean separation between the two systems.

CRM Extensions Are Complex but Necessary

Creating the "API Invoice Payers" window in the CRM as a new feature was more complex than anticipated. However, this dedicated interface provided enormous value for operations teams managing agency relationships. The investment paid off in operational efficiency and clear separation of concerns.

Business Rules Drive Technical Architecture

Business requirements for flexible pricing models and payment method filtering shaped the entire system design. Understanding business drivers before diving into technical implementation prevented misalignments between business expectations and technical delivery.

Backward Compatibility is Harder Than Greenfield

Building a new API from scratch allowed for clean architecture while using existing systems as reference. Creating separate capabilities and data models for invoice payments enabled focused development without the constraints of modifying live systems. This approach provided flexibility while maintaining operational stability.
