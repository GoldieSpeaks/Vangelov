Annual Data Deletion Exercise
Executing a GDPR-compliant annual purge of personally identifiable information across an education payments ecosystem — covering multiple account types with full audit trails and zero operational disruption.

As part of ongoing GDPR compliance and data security best practices, this project executed the annual deletion of personally identifiable information (PII) across the TransferMate Education ecosystem. The exercise targeted all user accounts inactive for more than 10 years, spanning multiple distinct account types across the platform's various payment solutions and applications.

Executive Summary

All qualifying PII was systematically removed from the production database with comprehensive deletion logs, QA verification on both test and live environments, executive sign-off, and audit-ready documentation — all completed without impacting day-to-day platform operations.

The resulting audit package was subsequently used to successfully defend the company's data handling practices during a compliance audit.

Project Scope

In Scope

  • All external-facing user account types (inactive 10+ years)
  • Internal portal and institutional accounts
  • Partner and sub-partner accounts
  • Associated PII: transactions, logs, refund history, payment processor data, beneficiaries, bank details
  • Reusable script creation for all current and newly added tables

Out of Scope

  • Back Office database (separate exercise)
  • SSO-integrated partner accounts (recent, not yet eligible)
  • API Dashboard accounts (created recently)
  • Invoice API accounts (created recently)
  • Institutional link records (company data, retained for sales)
  • Internal test accounts (reserved usernames)

Lessons & Insights
What Worked Well

Phased Approach by Account Type

Processing each account type independently with dedicated scripts and logs allowed isolated verification. If any phase encountered issues, it would not block others or compromise data integrity.

What Worked Well

Dual-Environment Verification

Running deletions first against a test database snapshot, then repeating on production after QA sign-off, provided confidence that the scripts behaved identically in both environments before touching live data.

Challenge

Script Evolution for Schema Changes

New tables and columns added throughout the year required careful analysis to extend the reusable deletion scripts. Without this annual review, orphaned PII data could persist in newly created tables, creating compliance gaps.

Key Learning

Data Corrections as Prerequisite

Inconsistencies in user product records had to be corrected before deletion could proceed safely. This highlighted the importance of data quality checks as a standard pre-deletion step.

Risk Mitigated

Off-Peak Execution Windows

Large-scale deletion operations were executed via shell console during off-peak hours to avoid Apache gateway timeouts and minimize server load impact on active users.

Key Learning

Comprehensive Audit Packaging

Consolidating all deletion logs, reports, and sign-off documentation into a single auditable package ensured readiness for compliance reviews without requiring additional effort after the fact.
