Regional Payment Provider Integration (Mexico)
Migrating bank transfer payments from Mexico to a regional payment provider — eliminating international fees, resolving customer complaints, and expanding collection coverage across web and API payment channels.

Bank transfer payments originating from Mexico were processed through a US-based bank account, incurring additional international fees that led to increasing customer complaints. A previously held local account had been closed, leaving the platform without a cost-effective collection solution for the region. The team integrated a regional payment provider to handle all future Mexican bank transfer payments via redirect-based collection, covering both web platform and API channels within a compressed 7-week timeline. This integration subsequently became the reusable template for six additional regional payment corridors.

6 Overview
Executive Summary

Bank transfer payments originating from Mexico were processed through a US-based bank account, incurring additional international fees that led to increasing customer complaints. A previously held local account had been closed in October, leaving the platform without a cost-effective collection solution for the region.

The team integrated a regional payment provider to handle all future Mexican bank transfer payments via redirect-based collection. The integration covered both web platform and API channels within a compressed 7-week timeline. Margins were set at 3.5% to maintain profitability after rate negotiations, and multiple client exceptions were carefully managed to ensure backward compatibility with legacy integrations.

This project proved particularly significant beyond its immediate scope: the redirect-based collection pattern became the reusable template for six additional regional payment corridors, establishing a scalable approach to market expansion.

8 Scope
Project Scope

In Scope

  • Web platform — Payer logins redirected to regional provider
  • API — New version clients receive redirect URL in response
  • API — Anonymous payer flow via redirect
  • Margin configuration (3.5% for MXN via provider)
  • Settlement in EUR/USD based on beneficiary currency
  • Exception handling for four legacy client categories

Out of Scope

  • Legacy API clients (no redirect support in older versions)
  • Platform agents (incompatible with redirect process)
  • Partner system payments (not available in region dropdown)
  • Same-currency fee handling (flagged post-release)
  • Bulk agent payment adjustments (deferred)

12 Learnings
Lessons & Insights
● What Worked Well

Compressed Delivery Timeline

From problem identification to production go-live in approximately 7 weeks. The team maintained quality while moving fast, with both internal and production testing achieving 100% pass rates across 22 test cases.

● What Worked Well

Reusable Integration Template

The redirect-based collection pattern established in this project was subsequently reused for six additional regional payment corridors (Colombia, Peru, Chile, Indonesia, Pakistan, Thailand), proving the architecture's scalability.

● Challenge

Rate Negotiation Complexity

The provider's initial rate of 3.2% exceeded the platform's previous 2.2% via the local bank. Securing profitability required setting a 3.5% margin for payers — a significant increase that required careful stakeholder communication.

● Key Learning

Legacy API Compatibility Constraints

Older API clients cannot support redirect-based payment flows. This architectural limitation means new payment provider integrations are only available to clients on newer API versions, creating a two-tier experience that requires explicit exception handling.

● Risk Identified

Same-Currency Fee Gap

No additional fee was configured for same-currency payments (MXN-to-MXN). Identified post-go-live by the development team. Flagged for remediation but represents a potential revenue gap that should have been caught in specification review.

● Reflection

What I'd Do Differently

Would have included same-currency edge cases in the original BRD acceptance criteria. The specification focused on cross-currency scenarios (MXN to EUR/USD) but missed the edge case where both source and destination are MXN. A more comprehensive fee matrix review before sign-off would have prevented this post-release gap.
