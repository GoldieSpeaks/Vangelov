Tax Collection API Integration
Integrating a mandatory tax calculation API into the payment flow for a regulated corridor — eliminating secondary payment requests, reducing partner operational overhead, and ensuring tax compliance is handled seamlessly within a single transaction.

Regulatory requirements mandate that Tax Collected at Source (TCS) is applied to all outbound payments from a specific jurisdiction. Previously, the regional collection partner handled tax separately — requiring payers to make a second payment outside the platform. This created poor user experience, operational burden on the partner, and compliance risk. This project integrated the partner's tax calculation API directly into the payment flow, covering both web and API channels with graceful degradation on API failure.

6 Overview
Executive Summary

This project integrated a regional collection partner's Tax Collected at Source (TCS) API into the payment platform, covering both web payment links and the client-facing API. The integration enables real-time tax calculation as part of the payment flow, eliminating the need for payers to make secondary payments and removing significant manual processing from the partner's operations.

The project required careful negotiation with the partner (who rejected an internal calculation approach), dual-channel development across web and API surfaces, a grace period strategy for API clients to adopt new mandatory parameters, and graceful degradation messaging for API failure scenarios. A full RAID log and security risk assessment were completed pre-release.

The integration was released to production with 14 sub-task deliverables across both channels. Post-release, a decimal validation fix and corridor exclusion refinement were shipped within days, demonstrating rapid iteration on edge cases identified in production.

12 Learnings
Lessons & Insights
● What Worked Well

Grace Period Strategy

Introducing new API parameters as non-mandatory initially gave clients a transition window without breaking existing integrations. This approach enabled compliance enforcement on a timeline rather than creating a hard cutoff that could have disrupted active payment flows for dozens of API clients.

● What Worked Well

Proactive Client Communication

Drafting and sending a structured notification email before the changes went live gave API clients advance notice of the new parameters. This prevented support escalations and demonstrated professional change management to the partner ecosystem.

● Challenge

Partner Rejection Required Pivot

The initial proposal to calculate tax internally was rejected by the partner, requiring a complete pivot to their API integration approach. This added complexity (sequential API calls, failure handling, timeout scenarios) that would not have existed with internal calculation. Early alignment on integration approach would have saved analysis time.

● Key Learning

Corridor Exclusion Is Critical

Post-release discovery that tax was being applied to non-partner payment links for the same currency highlighted the importance of routing-level exclusion logic. The fix was shipped within days, but the edge case should have been identified during specification review rather than in production.

● Risk Identified

Partner API Reliability Dependency

The platform's tax accuracy is now dependent on the partner's API availability. While fallback messaging handles outages gracefully, extended downtime could impact payer experience at scale. The tcs_api_fail database flag enables monitoring, but a long-term internal calculation fallback may be needed.

● Reflection

What I'd Do Differently

Would have pushed harder for a hybrid approach during initial partner negotiation — using their API as primary but maintaining an internal calculation as fallback. The current architecture has a single point of failure on the partner's API. A cached rate approach with periodic refresh could provide resilience without violating the partner's requirement for API-based integration.
