Direct Debit Automation
Replacing a fully manual direct debit refund process with automated payment method determination, beneficiary-level storage, and multi-currency consolidated reporting — eliminating error-prone email-based workflows across the payments operations team.

The treasury team identified that API-channel direct debit refunds were processed entirely through manual email communication. There was no system indicator distinguishing direct debit from bank transfer refunds, no stored DD account details, and no reporting capability. This project automated all three gaps within a single quarter, delivering database-driven logic that triggers on refund creation and routes payments automatically based on beneficiary configuration.

6 Overview
Executive Summary

The direct debit refund process for API-integrated clients was entirely manual: the payments team relied on email requests to determine whether refunds should be collected via direct debit or bank transfer. There was no system indicator, no stored DD details, and no reporting capability. The treasury team raised this as a priority due to error risk and time consumption.

This project automated all three gaps. A database trigger now fires on refund line creation, checking the beneficiary's direct debit flag to automatically set the refund payment method. Direct debit account details are stored at the beneficiary level with role-based access controls restricting edits to authorised personnel. A consolidated CSV report enables the payments team to export all DD refund lines across four currencies within any date range.

The project was deployed and validated in the live environment since the functionality was entirely new with no existing production data to test against. All QA defects were resolved, permissions validated in production, and the system is now fully operational.

12 Learnings
Lessons & Insights
● What Worked Well

Pragmatic Production-First Testing

The decision to validate in production was not avoidance of process but a pragmatic choice: the DD functionality had no existing data in test environments. Testing with real transaction data ensured meaningful validation rather than synthetic scenarios that would have missed the GBP report issue entirely.

● What Worked Well

Parallel Development Strategy

Starting the core trigger logic immediately (resolved in 3 weeks) while the full BRD was being authored allowed the project to deliver faster. The trigger was production-ready before the reporting and storage features even entered development, de-risking the most critical piece early.

● Key Learning

Report Requirements Need Explicit Scope

The GBP filtering issue revealed that the BRD specified a "consolidated report" but did not clearly define whether per-currency filtering was expected. QA reasonably assumed individual currency reports were possible. Explicit scope boundaries in reporting specifications prevent this class of defect.

● Challenge

Security Controls Must Be Day-One Features

The permission bypass (DD editable by unauthorized users) was caught during QA testing. For financial data like direct debit account details, role-based access controls should be baked into the initial specification and tested from the first build, not discovered as a defect.

● Risk Identified

Blank State Validation Gaps

The ability to save a blank refund payment method was a data integrity risk. While the trigger handles new lines correctly, manual overrides could still introduce null states. This edge case would have been caught earlier with explicit null-state testing in the test plan.

● Reflection

What I'd Do Differently

Would have included explicit permission test scenarios in the BRD from day one, rather than relying on QA to discover the access control gap. For any feature touching financial data, a "permissions matrix" table in the spec would have caught the super-user-only requirement before code was written.
