Enterprise API Development
Building a bespoke RESTful API suite for a top-tier business school — enabling automated payment plan management, batch processing, and bi-directional data synchronization with their payer information system.

The enterprise client required programmatic access to manage payment plans, installments, and payer accounts on the cross-border payment platform. The API supports batch insert, update, and retrieval operations with JWT-based authentication using asymmetric RS256 signing. Four dedicated environments were provisioned with CI/CD pipelines, and comprehensive OpenAPI-compliant documentation was published for the client's integration team. This was a high-revenue engagement with a contracted delivery agreement targeting a specific academic intake cycle.

6 Overview
Executive Summary

This project delivered a custom API suite for a top-tier business school, enabling their payer information system to programmatically manage payment plans, installments, and payer accounts on the cross-border payment platform. The API supports batch insert, update, and retrieval operations with JWT-based authentication using asymmetric RS256 signing.

Four dedicated environments (DEV, INT, UAT, PRD) were provisioned with CI/CD pipelines. The client's integration team began testing against the INT environment while core development continued in parallel — a pragmatic approach that compressed the overall timeline. A critical batch processing bug was identified and resolved same-day during joint testing.

The enterprise client can now automate the creation and management of payment plans for thousands of payers across multiple campuses, programs, and currencies — with comprehensive OpenAPI documentation enabling their team to integrate independently.

7 Delivery
Project Timeline

6-month development lifecycle from JSON structure analysis through production deployment and client validation.

Weeks 1-2
JSON Structure Analysis & Field Mapping
Analyzed the client's payer information system payload structure. Product Manager created comprehensive field mapping document translating client schemas into platform-compatible data models for payment plans, installments, and payer/sponsor accounts.
Week 3
Environment Setup & JWT Authentication
Provisioned 4 API environments with CI/CD pipelines. Implemented JWT authentication with RS256 asymmetric signing. Per-environment API users created with linked sub and aud claims ensuring complete environment isolation.
Weeks 4-6
Insert Endpoint Development
Built batch insert endpoint for payment plans with nested student/sponsor account creation. Complex JSON payload handling supporting multi-installment plans with per-record validation and independent success/failure reporting.
Weeks 7-8
Update Endpoint & Preferred Email Feature
Developed batch update endpoint with per-record error handling. Added preferred email notification field based on client feedback during iterative design sessions. Feature introduced proactively by the development team.
Weeks 9-12
INT Environment Deployment & Client Testing
Deployed fully working insert, update, and retrieve functionality to INT environment. Client began integration testing against their payer information system. Pragmatic decision to deploy before internal QA to enable parallel testing.
Week 13
Batch Update Hot Fix
Critical bug identified where only the first record in a batch update processed successfully. Same-day hot fix deployed and validated by the development team. Issue resolved without requiring formal QA cycle due to urgency.
Weeks 14-16
Retrieval Endpoint & Advanced Filtering
Built flexible retrieval endpoint with 20+ filter dimensions, configurable matching modes (IN, BETWEEN), and combinable AND/OR logic. Enables the client to query payment plan data across any combination of program, campus, status, and date ranges.
Weeks 17-20
Internal QA Validation
QA engineer validated all endpoints across internal and external environments. Insert, update, and retrieve functionality tested with valid, invalid, and boundary condition scenarios. All test cases passed with no issues found.
Weeks 21-22
Full Environment Deployment
All API environments deployed to production. Client confirmed successful JWT authentication process across all environments. Release sign-off obtained and live status achieved.
Weeks 23-24
OpenAPI Documentation & Handover
Comprehensive OpenAPI 3.0+ Swagger documentation published covering all endpoints, request/response schemas, authentication methods, error codes, and example payloads. Swagger UI accessible for the client's developer self-service reference.

10 Decisions
Key Decisions
When Decision Type
Week 1 JWT RS256 over symmetric signing. Asymmetric key pair chosen for enterprise security requirements, ensuring the platform never holds the client's private key. Technical
Week 3 Per-environment API users with isolated credentials. Each environment (DEV/INT/UAT/PRD) gets its own API user and key pair to prevent cross-environment access. Technical
Week 4 Per-record error handling in batch operations. Rather than fail-all-on-one-error, each record is independently validated and processed, returning granular success/failure per item. Process
Week 5 Refunds excluded from MVP scope. Too much development required; would exceed contracted timeframes. Deferred to subsequent phase to protect core delivery. Strategic
Week 9 Early deployment to INT before internal QA. Client needed to test against their systems, so endpoints were deployed externally before completing internal QA — a pragmatic approach requiring careful coordination. Strategic
Week 21 OpenAPI 3.0+ standard for documentation. Swagger documentation chosen as the industry standard, providing the client's developers a self-service reference and reducing back-and-forth communication. Process

12 Learnings
Lessons & Insights
● What Worked Well

Environment-First Approach

Provisioning all four environments upfront enabled the enterprise client to begin integration testing early while development continued in parallel. This compressed the overall timeline significantly and gave the client confidence in the platform's readiness.

● What Worked Well

OpenAPI Documentation Standard

Publishing Swagger documentation early gave the client's developers a self-service reference, reducing back-and-forth communication and enabling parallel development on both sides. The documentation became the single source of truth for integration questions.

● Challenge

Commercial Expectations vs. Reality

Delivery timelines were set commercially before technical scoping was complete, creating pressure that required active management. Iterative JSON structure changes from the client added complexity that wasn't accounted for in original estimates.

● Key Learning

Pre-Release Client Testing Accelerates Delivery

Deploying endpoints to the external environment before internal QA completion was a calculated risk that paid off. It enabled the client to validate integration patterns early, surfacing the batch update bug during joint testing rather than post-launch.

● Risk Identified

Batch Processing Edge Cases

The critical bug where only the first record in a batch update processed successfully highlighted the need for multi-record batch testing as a standard practice. Same-day resolution prevented client impact, but earlier automated batch testing would have caught this sooner.

● Reflection

What I'd Do Differently

Would have pushed for direct communication channels with the client's technical team from day one. Routing all technical decisions through commercial intermediaries created delays and miscommunication that added unnecessary friction to the delivery process.
