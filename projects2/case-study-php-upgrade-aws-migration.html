PHP 8 Upgrade & AWS Cloud Migration
Coordinating a dual-track infrastructure transformation — upgrading the platform runtime to PHP 8 while simultaneously migrating hosting to AWS — across a large cross-functional team through four release target adjustments.

The platform ran on an outdated PHP version hosted on traditional infrastructure, creating mounting security exposure, performance limitations, and incompatibility with modern libraries. The decision was made to combine the PHP 8 upgrade with a simultaneous migration to AWS, consolidating two major infrastructure changes into a single coordinated initiative. This dual-scope project involved multiple development teams, QA resources, DevOps specialists, and project leadership across a six-month delivery timeline that required four release date adjustments as complexity was uncovered.

6 Overview
Executive Summary

The platform operated on an end-of-life PHP version hosted on traditional infrastructure. Security patches were no longer available for the runtime, modern libraries required PHP 8 as a minimum, and the hosting environment lacked the scalability and automation capabilities of cloud platforms. Both issues needed addressing, and the decision was made to combine them into a single initiative to minimise total disruption.

The project involved upgrading the entire codebase to PHP 8 compatibility, migrating all hosting infrastructure to AWS, and validating that the full platform functioned correctly on both the new runtime and new infrastructure simultaneously. A cross-functional team of nine worked across development, DevOps, QA, and project coordination, with the release target adjusted four times as the true scope of compatibility work became apparent.

The platform was successfully migrated to AWS running PHP 8, with full regression validation confirming zero functional impact. The combined approach saved months of repeated testing that sequential migrations would have required.

7 Delivery
Project Timeline

Six-month delivery from epic creation through iterative release adjustments to successful production deployment.

Month 1
Epic Creation & Strategic Scoping
Epic created defining the combined PHP 8 upgrade and AWS migration as a unified infrastructure initiative. Initial release target set for March. Cross-functional team identified: developers, DevOps, QA, and project coordination resources.
Month 1-2
Team Assembly & Work Breakdown
Nine team members assigned across parallel workstreams. PHP compatibility assessment began on the codebase while DevOps started AWS environment provisioning. Child tasks created for each major component requiring PHP 8 adaptation.
Month 2
First Release Adjustment (March → April)
PHP compatibility analysis revealed more extensive refactoring needed than initially estimated. Deprecated function usage was pervasive across the codebase. Release target moved to April to accommodate the full scope of compatibility work.
Month 3
Core Development & AWS Setup
PHP 8 codebase adaptations progressed across all major modules. AWS infrastructure configured with matching environment specifications. Integration challenges between new PHP version and existing database layer identified and resolved.
Month 3
Second Release Adjustment (April → May)
AWS migration complexity exceeded initial estimates. Environment parity between legacy hosting and AWS required additional configuration work. QA flagged that full regression testing needed more time given the dual-change scope.
Month 4
Integration Testing & Third Adjustment (May → June)
Combined testing of PHP 8 code on AWS infrastructure began. Edge cases emerged in specific platform modules requiring additional developer attention. Release moved to June to ensure comprehensive regression coverage.
Month 5
Full Regression & Performance Validation
Complete platform regression testing on AWS with PHP 8. Performance benchmarking confirmed improvements over legacy stack. Load testing validated elastic scaling behaviour under production traffic patterns.
Month 5-6
Pre-Production Hardening
Final hardening phase: security configuration, monitoring setup, rollback procedures documented, and cutover plan established. Team rehearsed migration procedure to minimise production transition window.
Month 6
Production Deployment
Platform migrated to AWS running PHP 8 during planned maintenance window. Cutover completed within scheduled timeframe. Immediate post-deployment monitoring confirmed all systems operational with no functional regression.
Month 6+
Post-Deployment Validation & Closure
Extended monitoring period confirmed platform stability on new infrastructure. Performance metrics validated against baseline. Project formally closed and marked as Live. Infrastructure modernisation roadmap unblocked for future enhancements.

12 Learnings
Lessons & Insights
● What Worked Well

Combined Initiative Approach

Running the PHP upgrade and AWS migration as a single programme saved months of duplicated regression testing. The team only needed to validate the full platform once on the combined new stack, rather than twice for each change independently.

● What Worked Well

Flexible Release Scheduling

Adjusting the release target four times showed willingness to prioritise quality over deadlines. Each adjustment was made when real complexity was uncovered, preventing a rushed deployment that could have caused production incidents on critical infrastructure.

● Challenge

Scope Estimation Complexity

The initial timeline significantly underestimated the PHP compatibility work required. Deprecated function usage was more pervasive than code analysis suggested. Future infrastructure upgrades should include a dedicated compatibility assessment phase before committing to timelines.

● Key Learning

Cutover Rehearsal Value

Rehearsing the production migration procedure before the actual cutover identified timing issues and procedural gaps that would have extended the maintenance window. This practice should be mandatory for all infrastructure migrations affecting production traffic.

● Risk Managed

Dual-Change Production Risk

Changing both runtime and hosting simultaneously doubled the potential failure surface. The rehearsed rollback procedure, comprehensive regression coverage, and planned maintenance window with full team availability created the safety net needed for a confident deployment.

● Reflection

What I'd Do Differently

Would have invested in a dedicated compatibility assessment phase upfront before committing to the initial release target. A two-week deep analysis of deprecated PHP function usage across the codebase would have produced a more realistic timeline from the start and avoided the perception of repeated delays.
