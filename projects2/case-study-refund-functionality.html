Refund Functionality
Building a comprehensive end-to-end refund processing capability across web payment platforms, API systems, and back-office operations — enabling clients to request, track, and manage refunds through a unified workflow.

Previously, refund requests had limited automation and lacked proper status tracking across a cross-border education payments ecosystem. This initiative introduced refund request capabilities through both web and API channels, with changes flowing into operations systems for processing. The project spanned 12 deliverable areas across 3 platform layers, delivering webhook notifications, sandbox testing support, compliance-ready transaction types, and integrated reporting over a multi-quarter timeline.

6 Overview
Executive Summary

This project delivered a comprehensive refund processing system across the entire education payments platform. Previously, refund requests had limited automation and lacked proper status tracking. The initiative introduced refund request capabilities through both web payment platforms and API systems, with changes flowing into the back-office systems for operations teams to process.

Key deliverables included database schema improvements for proper refund tracking, a refund status lifecycle engine with distinct transaction types for compliance screening, webhook notifications for real-time status updates, sandbox refund support for client testing, and integrated reporting with refund history views.

The project resolved a critical data integrity issue where zero-value transaction amounts required a workaround using paid amounts as fallback — ensuring validation logic could correctly enforce refund amount limits against original transaction values.

12 Learnings
Lessons & Insights
● What Worked Well

Phased Delivery Approach

Breaking the refund capability into distinct deliverables (request flows, status management, webhooks, sandbox, reporting) allowed incremental value delivery and reduced integration risk across a multi-quarter timeline.

● What Worked Well

Scope Discipline

Deliberately excluding email notifications and cancellation requests kept the project focused on core refund flow delivery. The gating decision prevented feature creep while delivering maximum value from the approved scope.

● Challenge

Data Integrity Issues

Discovery of zero-value online_amount fields across thousands of transactions required a pragmatic workaround using paid amount as fallback. This highlighted the importance of data quality audits before building validation logic on existing fields.

● Key Learning

Compliance-Driven Design

The requirement for separate transaction types and distinct classifications for refund split lines ensured compliance screening could properly differentiate refund flows from standard transactions across education, corporate, and regional screening processes.

● Risk Identified

Cross-System Timeline Dependencies

Integration with corporate systems required extending the timeline from Q1 to Q3. Multi-system dependencies are inherently unpredictable and need buffer built into initial planning rather than accommodated reactively.

● Reflection

What I'd Do Differently

Would have conducted the data quality audit on transaction amounts earlier in the scoping phase rather than discovering the zero-value issue mid-development. Earlier identification would have allowed the team to design the validation logic correctly from the start rather than implementing a workaround.
