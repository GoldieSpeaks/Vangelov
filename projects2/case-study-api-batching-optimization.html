6 Overview
Executive Summary

This project introduced a new batching functionality within the back-office system to manage bulk API payment processing. The system automatically groups transactions into batches based on three matching criteria (beneficiary identity, account number, and currency), assigns unique batch identifiers, and enforces validation rules before payments can be transferred to downstream platforms.

The developer estimated 15-25 days of development effort. The system was delivered within 6 weeks end-to-end, passing internal QA with zero issues. A production defect was discovered during live testing (column reference error when restricting test data on the production database) and resolved in a subsequent fix.

The feature was later enhanced with a Phase 2 that expanded batching to include secondary payment lines with standard transfer cases and improved the transaction ID input for paste operations — driven by follow-up calls identifying real-world usage patterns that the initial specification did not cover.

12 Learnings
Lessons & Insights
● What Worked Well

Clean Internal QA Results

Internal testing on the test environment passed with zero issues across all batch creation, grouping, filtering, and transfer scenarios. This demonstrated thorough development adherence to specifications and gave confidence for production deployment.

● What Worked Well

Rapid Delivery Cadence

From specification to production in 6 weeks (with 15-25 days of active development). The project moved through requirements, development, QA, and release without blockers or scope disputes — enabled by clear specification from day one and immediate developer assignment.

● Key Learning

Environment-Specific Code Paths Need Testing

The production defect (column "t.is_test" not found) occurred because the test data restriction logic was only relevant on the production database. Code that behaves differently across environments must be tested in all target environments, not just the development one.

● Challenge

Exclusion Rules Need Clear Documentation

The Phase 2 edge case (admin fee lines being incorrectly included in batches) showed that exclusion criteria must be explicitly documented and tested. The initial "transfer case Regular" requirement was clear, but its implications for split lines were not obvious until real usage surfaced the gap.

● Risk Identified

Cross-Platform Status Dependency

Real-time synchronisation between back-office and web platforms introduces a system dependency. If either platform experiences downtime, status mismatches could cause processing delays or conflicting batch states that require manual reconciliation.

● Reflection

What I'd Do Differently

Would have included production-specific test scenarios in the QA plan from the start, particularly for code paths that reference environment-specific columns or configurations. A simple "run on production-like data" step would have caught the is_test column issue before go-live rather than two months later.
