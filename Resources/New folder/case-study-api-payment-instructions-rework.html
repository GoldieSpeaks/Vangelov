Case Study

API Payment Instructions Rework

Transforming raw, unstyled HTML payment instructions into self-contained, consistently rendered content across all API integration partners — standardizing structure, presentation, and compliance across multiple regions and languages.

E Executive Summary

The cross-border payments platform's API returned bank transfer instructions as raw HTML without any embedded styling. API partners received this HTML and rendered it on their own surfaces — often with their own CSS or none at all. Payers were presented with large, unstyled blocks of text at the most critical point in the payment journey: the moment they needed to complete a bank transfer. The lack of visual hierarchy made warnings indistinguishable from general text, risking payment errors and abandonment on high-value transactions.


Beyond the presentation problem, the instruction sets had grown organically and suffered from inconsistent element ordering, missing critical elements (the payment reference instruction was absent from most sets), outdated security wording, and country-specific content scattered throughout rather than clearly separated. This project delivered self-contained HTML with embedded CSS, a standardized 8-element structure across all sets, updated compliance language, and clean separation of regional content — giving the platform control over payer experience for the first time.

💡 Lessons & Insights

What Worked Well

Self-Contained HTML Eliminates Rendering Dependency

By embedding styles directly in the HTML content, the platform gained control over payer experience for the first time. Instructions now render correctly regardless of the API partner's own CSS or lack thereof.

What Worked Well

Standardized Structure Across All Sets

The 8-element order creates a consistent payer experience across all regions and platforms. One reference set serves as the template for all others, making future maintenance straightforward.

Challenge

Infrastructure Pause Caused Context Loss

A platform-wide infrastructure migration paused this project for several months. When work resumed, a full re-analysis was required because the original context had been lost — highlighting the cost of long project interruptions.

Key Learning

Regional Complexity Is a Product in Itself

Certain regions required multi-step fulfilment processes, partner integration steps, extensive document requirements, and bilingual content. Each regional variant is essentially a separate product that needs independent analysis and verification.

Technical Debt Identified

Missing Critical Payment Element

The payment reference instruction — essential for payment allocation — was missing from most instruction sets. This likely contributed to unallocated payments. The standardization effort surfaced and fixed this gap across all sets.

Key Learning

Phased Partner Rollout Reduces Risk

Since this changes the HTML structure partners currently consume, a phased rollout (sandbox first, then production per partner) ensures each integration is validated before go-live. This avoids a big-bang deployment that could break multiple partner surfaces simultaneously.
