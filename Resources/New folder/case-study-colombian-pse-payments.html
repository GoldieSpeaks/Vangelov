Regional Bank Transfer Integration (Colombia)

Integrating a regional electronic bank transfer payment method for Colombia across multiple payment channels — replacing an existing provider, enabling invoice-based bulk payments, and opening new revenue corridors in the Latin American market.

E Executive Summary

The cross-border payments platform needed to enable bank transfer payments from Colombia using a widely adopted regional electronic payment method. The existing provider had limitations — it could not support invoice-based bulk payments where individual payer details are not collected, and the platform sought a more cost-effective partnership.


The team integrated a regional payment provider using a redirect-based flow to handle all Colombian bank transfer payments across web payment links, payer API, and the invoice API. A parallel effort to separate bank transfer logic was required to accommodate the new payment method. The project involved coordination with external partners for sandbox credentials, settlement model alignment, and production validation with penny tests.

⚙ Technical Architecture

Redirect-based regional payment integration across three payment channels

☑ Project Scope

In Scope

  • Web payment links — Electronic bank transfer via redirect
  • Payer API — Redirect URL in payment response
  • Invoice API — Bulk payments with NIT (Tax ID) parameter
  • Operations — COP recalculations on active orders
  • Colombian GMF tax (0.4%) added to payment amounts
  • Bank transfer separation (parallel refactor for API logic)
  • Settlement model alignment (FX to USD operational account)

Out of Scope

  • Partner cross-border system payments
  • Agent payment links
  • Legacy API users (no redirect support)
  • Rate of exchange display on user interface

💡 Lessons & Insights

What Worked Well

Provider Redirect Simplification

The provider's decision to handle bank details collection on their hosted page significantly reduced integration complexity. The platform only needed to pass core payment parameters and handle the redirect flow.

What Worked Well

Bundled Bank Transfer Separation

Combining the bank transfer separation refactor with this project ensured the API architecture was future-proof. This parallel effort enabled cleaner integration for subsequent regional payment corridors.

Challenge

External Dependency on Credentials

Waiting on production credentials from the regional provider introduced delays. Sandbox testing progressed but production validation (penny tests) could only happen after credentials were provided, creating a bottleneck in the final weeks.

Challenge

Settlement Model Alignment

Determining the settlement currency and account required coordination between treasury, the commercial team, and the provider. This cross-team alignment took several weeks to finalize and blocked BRD completion.

Post-Release Finding

Bank-Specific Payment Issues

After go-live, several payers experienced payment failures with a specific Colombian bank. Investigation was initiated with the provider, requiring coordination across payments, operations, and the external partner.

Key Learning

Regional Tax Complexity

Colombia's 0.4% GMF tax requirement was discovered during scoping and had to be factored into the payment calculation. Regional payment integrations must always account for local tax requirements that may not be obvious from API documentation alone.
