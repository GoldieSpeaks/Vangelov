Regional Payment Provider Integration (Mexico)

Migrating bank transfer payments from Mexico to a regional payment provider — eliminating international fees, resolving customer complaints, and expanding collection coverage across web and API payment channels.

E Executive Summary

Bank transfer payments originating from Mexico were processed through a US-based bank account, incurring additional international fees that led to increasing customer complaints. A previously held local account had been closed, leaving the platform without a cost-effective collection solution for the region.


The team integrated a regional payment provider to handle all future Mexican bank transfer payments via redirect-based collection. The integration covered both web platform and API channels within a compressed 7-week timeline. Margins were set to maintain profitability after rate negotiations, and multiple client exceptions were carefully managed to ensure backward compatibility with legacy integrations.

💡 Lessons & Insights

What Worked Well

Compressed Delivery Timeline

From problem identification to production go-live in approximately 7 weeks. The team maintained quality while moving fast, with both internal and production testing achieving 100% pass rates.

What Worked Well

Clear Exception Handling

Four client exception categories were identified early and documented explicitly. Legacy clients continued on the existing path without disruption, while new clients benefited from the integration immediately.

Challenge

Rate Negotiation Complexity

The provider's initial rate of 3.2% exceeded the platform's previous 2.2% via the local bank. Securing profitability required setting a 3.5% margin for payers — a 60% increase that risked agent dissatisfaction.

Post-Release Finding

Same-Currency Fee Gap

No additional fee was configured for same-currency payments (MXN-to-MXN). Identified post-go-live by the development team. Flagged for remediation but represents a potential revenue gap.

Key Learning

Legacy API Compatibility Constraints

Older API clients cannot support redirect-based payment flows. This architectural limitation means new payment provider integrations are only available to clients on newer API versions, creating a two-tier experience.

Key Learning

Provider Integration as a Pattern

This project established the redirect-based collection pattern that was subsequently reused for other regional payment corridors (Colombia, Peru, Chile, Indonesia, Pakistan, Thailand) — proving the template's scalability.
