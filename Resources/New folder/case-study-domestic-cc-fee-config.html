Domestic Credit Card Fee Configuration

Introducing configurable percentage-based fees for same-currency card payments on web platforms — replacing rigid flat-fee structures with flexible, client-specific pricing.

E Executive Summary

Same-currency payments on web platforms were previously limited to bank transfers because the fixed fee structure did not cover card processing costs. Card and alternative payment method transactions were effectively blocked for domestic payments, reducing conversion rates and limiting payer choice. This project introduced percentage-based fee configuration for same-currency card payments, enabling the commercial team to set custom rates per client that ensure processing costs are covered while remaining competitive.


Key outcome: Web platform payers can now complete same-currency payments via credit card with dynamically calculated fees. The back-office system was updated to support the new fee model with dynamic text display. The implementation built on an existing API feature, extending it to web platforms with additional threshold logic to determine when percentage fees apply versus fixed fees.

⚙ Development Breakdown

Fee Calculation Engine

Core logic for applying percentage-based fees to same-currency card payments. Includes threshold detection: below a certain payment value, fixed fees remain more economical. Above the threshold, percentage-based fees kick in. Applied consistently for both payer and agent payments.

Operations Integration

Back office system updated to store and manage domestic payment fee percentages per client. This enables the payments and commercial teams to configure fee rates without developer intervention, supporting rapid onboarding of new fee structures.

Dynamic Fee Display

Checkout pages updated with dynamic text showing the calculated fee based on the payer's payment amount and configured percentage. This ensures transparency and reduces payment abandonment due to unexpected charges.

★ Lessons & Insights

What Went Well

Building on Proven API Logic

The API already had percentage-based fees, providing a validated reference implementation. This reduced design ambiguity and enabled the developer to focus on web platform adaptation rather than inventing new logic.

Learning

Fee Threshold Complexity

Determining the crossover point between fixed and percentage fees required business input. The threshold logic was not initially in the BRD and emerged from stakeholder calls, highlighting the need for explicit fee scenario modelling early in analysis.

Consideration

Exception Mapping Required

Existing same-currency fee configurations had client-specific exceptions and overrides. These needed to be mapped before development to avoid conflicts between existing flat fees and new percentage-based fees.

What Went Well

Dynamic Display Improves Trust

Showing payers the calculated fee before payment confirmation was a deliberate UX decision that improves payment completion rates. The dynamic text component is reusable for future fee display scenarios.
