E Executive Summary

This project introduced Google Pay and Apple Pay as digital wallet payment options on an international cross-border payments platform. By leveraging out-of-the-box wallet features from an existing payment gateway provider, the team delivered the capability with minimal custom code — focusing development effort on country-specific logo display, Apple Pay domain verification, and device-based dynamic switching between wallets. The project followed a structured lifecycle from initial discovery through business requirements documentation (BRD), executive sign-off, development, QA testing (both internal and production), and successful release.


Key outcome: Payers across supported countries can now use their preferred digital wallet at checkout, with the correct wallet option surfaced automatically based on device type — Apple Pay on Apple devices, Google Pay on Android. Both internal and external testing achieved a 100% pass rate across all test scenarios.

Deployed Apple Pay domain verification file (/.well-known/apple-developer-merchantid-domain-association) to all impacted domains. Required for Apple Pay to appear on provider's hosted forms.

File sourced from payment provider's domain. Must be deployed to any new domain added in future.

Dynamic Switching

Implemented device-based logic: Apple Pay visible on Apple devices only, Google Pay on Android only. Enforced mutual exclusivity — both must never appear simultaneously.

ACCEPTANCE CRITERIA
No overlap rule: both wallets must never be visible and selectable at the same time.

💡 Lessons & Insights

What Worked Well

Leveraging Existing Provider Capabilities

By choosing indirect integration via the payment gateway's hosted forms, the team avoided significant custom development. The wallets appeared automatically once enabled at the provider level, reducing the project to focused UI enhancements.

What Worked Well

Fast Executive Decision-Making

The country display decision was resolved in under 10 minutes by email. Both executives responded immediately with a clear, aligned answer. This set a benchmark for efficient stakeholder engagement.

Challenge

Parallel Workstream Coordination

Coordinating provider activation, BRD approvals, and development readiness required tight alignment across multiple teams. Proactive scheduling and clear ownership ensured no phase waited on another.

Key Learning

BRD Process as Quality Gate

The BRD went through 5 revisions across multiple contributors before reaching executive sign-off. While thorough, this 5-week process ensured complete clarity on use cases, requirements, and edge cases before any code was written.

Technical Debt Identified

Manual Domain Verification

The Apple Pay domain verification file must be manually deployed to any new domain. This creates a risk of Apple Pay silently failing on new domains if the deployment step is missed during onboarding.

Key Learning

Scope Discipline: Samsung Pay Deferral

When the payment provider raised Samsung Pay mid-project, the team correctly deferred it to a separate initiative rather than expanding scope. This kept the delivery timeline clean and avoided requirement creep.
