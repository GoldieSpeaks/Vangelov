14-Day Auto-Cancellation CRON Migration

Migrating a payment auto-cancellation process from application-level timers to a database-native scheduled job — reducing redundant traffic, improving execution frequency, and eliminating multi-instance duplication.

E Executive Summary

The cross-border payments platform had an existing process that automatically cancelled payment lines remaining in "Setup" status for 14 or more days. This process ran as a timer-based function inside the operations application, executing every 4 hours in every running instance of the application — generating significant unnecessary database traffic.


This project migrated the auto-cancellation logic to a database-native CRON job, eliminating application-level overhead entirely. The new implementation uses a materialized view to identify eligible records and a scheduled function to process cancellations every hour — a 4x increase in execution frequency with zero application traffic.


Key outcome: The migration was completed, tested on production, and validated by the QA team with all test cases passing. The solution also resolved the multi-instance duplication issue, ensuring the cancellation logic runs exactly once per scheduled interval.

</> Development Breakdown

Key implementation areas that delivered the migration

1

Materialized View

Created a SQL materialized view that identifies payment lines with "Setup" status persisting beyond the 14-day threshold. This pre-filters the dataset so the CRON function only processes relevant records.

PURPOSE
Reduce query load by materializing the eligible record set in advance.
2

CRON Function

Built the database function that processes records from the materialized view, transitioning payment line statuses from "Setup" to "Cancelled". Encapsulates the same business logic previously in the application.

BUSINESS RULE
If payment line status = "Setup" for ≥ 14 days → cancel automatically.
3

Scheduled Execution

Configured the database CRON scheduler to invoke the cancellation function every hour. Runs as a single database-level process regardless of how many application instances are active.

IMPROVEMENT
From 4-hour timer per instance → 1-hour single execution. No application involvement.

💡 Lessons & Insights

What Worked Well

Database-Native Over Application-Level

Moving the cancellation logic to the database level eliminated the multi-instance duplication problem entirely. The CRON job runs exactly once per interval, regardless of how many application instances are active.

What Worked Well

Clean Three-Part Architecture

The solution was cleanly separated into three components: materialized view (data), function (logic), and CRON schedule (trigger). This separation of concerns makes future maintenance straightforward.

Key Learning

Application Timers Scale Poorly

When the same timer runs in every application instance, the problem multiplies with scale. Background processes that don't require application context should be database-native from the start.

Challenge

Cross-System Status Verification

The cancellation in the internal system needed to correctly reflect on the web platform as well (Setup/Unsuccessful in one, Active/Inactive in the other). Testing required verification across both systems to confirm data consistency.

Key Learning

Frequency Improvements Come Free

Because the CRON approach is dramatically more efficient than multi-instance timers, the team could increase execution frequency from 4 hours to 1 hour while actually reducing total database load.

Consideration

Materialized View Refresh

The materialized view needs to be refreshed to reflect current data. The refresh strategy must balance data freshness against database performance, especially as transaction volume grows.
