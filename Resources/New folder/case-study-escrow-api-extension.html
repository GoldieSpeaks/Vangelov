Escrow Payments API Extension

Extending escrow fund-holding capabilities from web payment portals to the API system, enabling integrated partners to process visa-dependent individual payments through a single transaction flow.

S Executive Summary

The cross-border payments platform already supported escrow payments through its web portal for institutions requiring visa-dependent fund holding. However, partners integrated via the API system had no way to process escrow transactions, creating a service gap for institutions in regions with immigration compliance requirements. This project extended escrow capabilities to the API layer by adding new database fields, a conditional passport-country parameter, and country-level escrow flagging. A formal BRD was produced, cross-team alignment with payments and commercial stakeholders was conducted, and a phased delivery approach ensured the API component shipped first while the Back Office Systems extension followed separately. API documentation was also updated to reflect the new escrow parameters.

S Project Scope

In Scope

  • API escrow parameter (passport_issuing_country)
  • Beneficiary escrow flag (database column)
  • Country-level escrow requirement flag
  • Transaction escrow flagging logic
  • API documentation updates
  • Payment history escrow filter

Out of Scope

  • Web payment portals (already supported)
  • Partner/agent payment flows
  • Operations visa status management (separate phase)
  • Invoice API

L Lessons Learned

Phased Delivery Unblocked Partners Early

Shipping the API component independently from the Operations extension allowed partners to begin processing escrow transactions while the more complex internal workflows were still being built.

Cross-Team Discovery Prevented Rework

Investing time in scoping calls with the payments and treasury teams ensured the technical design aligned with actual fund-holding processes, avoiding costly post-development corrections.

Formal BRD Coordination Takes Time

The BRD cycle added several weeks to the timeline. For future projects with similar cross-team dependencies, earlier analyst engagement could compress the pre-development phase.

Conditional Mandatory Fields Reduce API Friction

Making passport_issuing_country mandatory only for escrow beneficiaries meant non-escrow API partners experienced zero disruption from the change.
