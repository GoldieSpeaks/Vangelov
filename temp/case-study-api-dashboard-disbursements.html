E Executive Summary

The current reconciliation process for API clients relies on manually exported reports from the back office system. These reports are prepared by the payments team and sent to clients, creating delays and operational overhead. Phase 2 of the API Dashboard introduces a Disbursements section that allows clients to view all payouts directly through a self-service web interface. The module aggregates both batch and individual payout data, provides filtering and date-range capabilities, and includes a transaction-level popup for detailed disbursement inspection.


Current status: The feature has been developed. Database schema has been created, BRD was signed off, and the main table/filter page, transaction popup, and cross-system data transfer handler have been delivered.

⚙ Development Breakdown

Database Schema

New database tables created in the web database. Schema designed to support both batch and individual payout records with full transaction-level detail.

Data Transfer Handler

Cross-system handler to receive disbursement data from the back office system and populate the web database. Ensures data consistency and supports incremental updates.

Main Dashboard Page

Client-facing disbursements page with filterable table showing all payouts. Includes date range selection, search capabilities, and summary views for reconciliation.

Transaction Popup

Drill-down popup displaying individual transactions within a disbursement batch. Provides the detail level clients need for accurate payment reconciliation.

★ Lessons & Insights (Preliminary)

Design Decision

Cross-System Data Architecture

Rather than querying the back office system in real-time, the decision was made to replicate disbursement data into the web database. This decouples the client dashboard from back office system availability and performance.

What Is Going Well

Building on Phase 1 Foundation

Phase 1 of the API Dashboard established the UI framework, authentication, and client access patterns. Phase 2 leverages this foundation, reducing setup time and ensuring consistent user experience.

Consideration

BRD Sign-Off Process

The BRD sign-off required multiple follow-ups and took longer than initially planned. For future phases, establishing a sign-off deadline in the project timeline could prevent delays in development kick-off.

Risk

Data Consistency Between Systems

Maintaining consistency between back office disbursement records and the web database requires careful handler design. Any sync failures could result in clients seeing incomplete payout information.
