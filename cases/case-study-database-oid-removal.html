5 Overview
Executive Summary

I identified that the platform's PostgreSQL database contained legacy Object Identifiers (OIDs) — a deprecated feature that newer PostgreSQL versions no longer support. These OIDs were embedded across database schemas, application queries, models, and API endpoints, creating a direct blocker for database version upgrades. I initiated this project because without removal, the infrastructure team could not modernise the database engine.

I commissioned a comprehensive technical analysis that mapped every OID usage across the entire system. I coordinated the systematic replacement of all OIDs with standard identifiers (IDs) across the database schema, application logic, and API layer, with development conducted on an isolated environment to prevent interference with parallel projects.

I released the project as a dedicated deployment, confirming zero regression across all affected systems. The database is now fully compatible with modern PostgreSQL versions, unblocking the infrastructure modernisation roadmap.

12 Learnings
Lessons & Insights
● What Worked Well

Isolated Development Environment

Using a separate development environment for the OID rework was critical. The changes touched a large number of interrelated files and systems, and isolating them prevented any disruption to other active projects during the multi-week development period.

● What Worked Well

Analysis-First Approach

Investing in a thorough technical analysis before any code changes reduced rework risk. The comprehensive audit document served as both a development guide and a testing checklist, ensuring no OID references were missed during replacement.

● Challenge

Legacy Debt Accumulation

OIDs had been embedded in the system for years without causing visible issues — until they became an upgrade blocker. This project highlights the value of periodically reviewing technical debt before it becomes a constraint on infrastructure modernisation.

● Key Learning

Dedicated Release Isolation

Shipping the OID removal as a standalone release made it possible to attribute any post-deployment behaviour directly to the OID changes. This isolation strategy should be the default for infrastructure-level modifications that touch every layer of the stack.

● Risk Managed

Cross-System Regression Risk

Replacing identifiers across every layer of the application stack carried significant regression risk. The staged testing approach — isolated environment, then main dev, then internal QA, then external QA — provided multiple safety nets that caught issues before production.

● Reflection

What I'd Do Differently

Would have pushed for an automated regression test suite before starting the rework. While manual QA caught everything, automated tests would have enabled faster iteration cycles and provided ongoing confidence as changes accumulated across the isolated environment.
