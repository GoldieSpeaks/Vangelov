Domestic Card Fee Configuration
I designed and delivered the percentage-based configurable card fee engine for same-currency payments, unlocking card transactions where previously only bank transfers were viable due to processing cost constraints.

I identified that same-currency domestic payments were limited to bank transfers because fixed processing fees made cards commercially unviable. I led stakeholder discovery, defined the fee model with threshold logic, and coordinated a cross-system release that enabled institutions to offer card payments with dynamically calculated and transparently displayed fees.

5 Overview
Executive Summary

I delivered a configurable percentage-based fee engine for domestic (same-currency) card payments on the web platform. The API equivalent had already been released for partner integrations, and I extended the capability to the institution-facing web interface.

I designed the fee engine to support per-client configuration with threshold logic — charging a percentage of the transaction value up to a configurable cap, beyond which a fixed fee applies. I ensured dynamic text display on the payment form shows payers the exact fee before they proceed, and integrated back office tooling for operations teams to view and manage fee configurations.

I coordinated the release alongside the bulk escrow payment deployment in a cross-system release, with both web platform and operations tooling going live simultaneously.

10 Decisions
Key Decisions
When Decision Type
Week 1 Percentage-based fees over fixed fees. Percentage model scales naturally with transaction size, covers processing costs proportionally, and is already proven in the existing API implementation. Strategic
Week 2 Cards only — exclude bank transfers from scope. Bank transfer fees have different cost structures and commercial models. Limiting scope to cards enabled faster delivery while the primary revenue opportunity was unlocked. Strategic
Week 3 Threshold cap: switch to fixed fee when percentage exceeds limit. For very large transactions, the percentage fee could become disproportionate. A configurable cap ensures fair pricing and prevents payer resistance on high-value payments. Technical
Week 4 Align calculation logic with existing API implementation. Consistency between API and web platform prevents discrepancies in fee amounts across channels. Shared logic reduces maintenance burden and testing surface area. Technical
Week 6 Real-time dynamic fee display on payment form. Showing the calculated fee as the payer enters their amount provides transparency, reduces abandonment, and eliminates post-payment queries about unexpected charges. Process
Week 10 Coordinated release with bulk escrow deployment. Both features required operations tooling changes. A unified release reduced deployment risk, minimized operations disruption, and allowed shared testing windows. Process

12 Learnings
Lessons & Insights
● What Worked Well

Existing API as Blueprint

Having the API implementation already live provided a proven calculation model, tested edge cases, and validated business logic. The web platform delivery could focus on presentation and configuration rather than inventing the fee model from scratch.

● What Worked Well

Coordinated Cross-System Release

Aligning this release with the bulk escrow deployment reduced deployment overhead, shared testing resources, and gave operations a single training window. The unified approach minimized disruption compared to two separate release cycles.

● Challenge

Threshold Logic Emerged Late

The need for a threshold cap (switching from percentage to fixed when the calculated fee becomes disproportionate) was not initially in the requirements. It surfaced through stakeholder calls when commercial teams raised concerns about high-value transaction scenarios.

● Key Learning

Fee Transparency Reduces Support Load

The dynamic display approach — showing the exact fee as the payer enters their amount — was validated as critical for reducing post-payment queries. Institutions that tested the feature reported significantly fewer fee-related support tickets than expected.

● Risk Identified

Exception Complexity Could Scale

The current exception mapping handles known scenarios (zero-fee agreements, special rates), but as more clients are onboarded with unique commercial terms, the priority logic may need restructuring. A rules engine approach would prevent combinatorial complexity.

● Reflection

What I'd Do Differently

Would model fee scenarios explicitly during the initial analysis phase rather than discovering threshold logic through stakeholder calls mid-project. Building a fee simulation spreadsheet showing calculations at various transaction sizes would have surfaced the cap requirement in week one rather than week three.
