Payment Integration
Local Payment Provider Replacement
I led the replacement of an underperforming third-party payment provider with a cost-effective local alternative, reducing transaction fees and accelerating settlement for a high-volume corridor across two platforms.

The existing provider for payments originating from a specific East Asian market was delivering slow processing, high fees, and poor settlement timelines. I replaced it with a local payment gateway offering static account routing, affecting both the web platform and API-integrated systems. I coordinated the external partner relationship, managed internal alignment calls, drove the demand management process, and ensured a penny test validated the integration before go-live.

5 Overview
Executive Summary

I replaced an existing third-party payment provider for a specific East Asian market corridor with a local payment gateway. The incumbent provider was delivering high transaction fees and slow processing times, directly impacting payment margins and client satisfaction. I targeted a single country corridor where the volume justified a dedicated local integration.

I designed an intentionally lightweight integration approach: deconstructing the existing provider framework and replacing it with static account routing for the new local gateway. Development was estimated at only a few days, but the coordination effort I managed — external partner alignment, DM request documentation, cross-team meetings, and QA validation — spanned several months from initiation to live deployment.

I successfully delivered the project through internal and external QA, and confirmed go-live with a penny test validating the production integration. Both the web platform and API systems were updated simultaneously.

10 Decisions
Key Decisions
When Decision Type
Week 1 Target single corridor only. I focused the replacement on one high-volume market to minimise risk while validating the new provider. Other corridors would be evaluated separately based on results. Strategic
Week 2 Static account routing over dynamic API integration. The new provider offered static account details, eliminating the need for complex API integration. Simplified development to a configuration change rather than a full integration build. Technical

12 Learnings
Lessons & Insights
● What Worked Well

Penny Test as Safety Net

Insisting on a penny test before enabling full traffic provided a critical safety net. The small validation payment confirmed the production integration worked correctly without risking real client payments. This approach should be standard for all payment provider changes.

● What Worked Well

Minimal Footprint Approach

Scoping the replacement to a single corridor with static account routing kept development effort minimal (days, not weeks). This focused approach de-risked the project and allowed rapid validation before considering expansion to other markets.

● Challenge

Governance Overhead vs Technical Simplicity

The DM request process took multiple weeks of scheduling attempts for what was technically a few days of development. The governance overhead significantly exceeded the implementation effort, though it served important compliance and documentation purposes.

● Key Learning

External Partner Coordination Requires Persistence

Aligning with external payment partners across different time zones and availability windows required persistent follow-up. Multiple rescheduled calls and repeated outreach were necessary before achieving the alignment needed to proceed.

● Risk Identified

Single-Provider Dependency per Corridor

Replacing one provider with another still leaves a single-provider dependency for the corridor. If the new provider experiences issues, there is no automatic fallback. A future consideration would be to maintain failover routing for critical payment corridors.

● Reflection

What I'd Do Differently

Would have initiated the DM request process in parallel with the external partner coordination rather than sequentially. The governance documentation could have been prepared concurrently, reducing the overall elapsed time from project creation to development assignment.
