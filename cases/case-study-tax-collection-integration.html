Tax Collection API Integration
I integrated a mandatory tax calculation API into the payment flow for a regulated corridor — eliminating secondary payment requests, reducing partner operational overhead, and ensuring tax compliance is handled seamlessly within a single transaction.

I led the integration of a regional collection partner's tax calculation API after identifying that the existing separate tax collection process created poor user experience, operational burden, and compliance risk. I negotiated the integration approach directly with the partner, coordinated dual-channel development across web and API surfaces, designed the grace period strategy for API clients, and ensured graceful degradation on API failure.

5 Overview
Executive Summary

I integrated a regional collection partner's Tax Collected at Source (TCS) API into the payment platform, covering both web payment links and the client-facing API. The integration I delivered enables real-time tax calculation as part of the payment flow, eliminating the need for payers to make secondary payments and removing significant manual processing from the partner's operations.

I negotiated directly with the partner (who rejected my initial internal calculation proposal), coordinated dual-channel development across web and API surfaces, designed a grace period strategy for API clients to adopt new mandatory parameters, and defined graceful degradation messaging for API failure scenarios. I completed the full RAID log and security risk assessment pre-release.

I released the integration to production with 14 sub-task deliverables across both channels. Post-release, I shipped a decimal validation fix and corridor exclusion refinement within days, demonstrating rapid iteration on edge cases identified in production.

7 Delivery
Project Timeline

End-to-end delivery from partner negotiation through dual-channel development, testing, and production release.

Week 1
Demand Request & Partner Negotiation
Business requested tax integration to eliminate secondary payment flow. I reached out to the regional collection partner proposing internal tax calculation. Partner rejected this approach, requiring full API integration with their system.
Weeks 2-3
Business Approval & Analysis Assignment
I escalated to business leadership for approval to proceed with the partner's API integration requirement. I assigned the Business Analyst to analyse the partner's API specification document and draft the functional specification.
Weeks 4-7
BRD & Specification (2 Versions)
I coordinated the BRD covering sequential API call logic (FX Rate, TCS, GST, Order), failure handling, and dual-channel requirements. I drove the update to version 2 incorporating the partner's revised specification document and obtained sign-off.
Week 8
Development Kickoff
I assigned the developer with a target completion date and allocated the QA engineer. Development began on web channel integration first, with new form fields for source of funds, declaration amount, and payer identification.
Weeks 9-12
WEB & API Integration Development
I oversaw the core TCS calculation integration into both WEB payment forms and API v3/v4 endpoints. The sequential API call chain was implemented: FX Rate query, TCS calculation, GST calculation, Order submission. I specified the database flag for API failure tracking.
Week 13
Failure Handling & Messaging
I conducted the business call to agree on API failure rate handling. I defined the fallback messaging strategy for payment instructions and payment selection screens when the partner's tax API is unavailable.
Week 14
Client Communication & Grace Period
I drafted the communication email to all API clients detailing new mandatory parameters. I defined the grace period: parameters introduced as non-mandatory initially, then enforced after the adaptation window. I updated API documentation.
Week 15
Payment Instructions Rework
I comprehensively reworked payment instructions for both channels. I specified tax-inclusive amounts displayed when API succeeds, disclaimer messaging for failure scenarios, and restructured existing instruction steps to incorporate tax information.
Week 16
QA Testing & Production Release
I coordinated QA testing across multiple environments and finalised the RAID log and security risk assessment. I released the feature to production in the April release window. All internal test scenarios passed.
Post-Release
Edge Case Fixes & Corridor Exclusion
I shipped two post-production refinements: a decimal validation fix for the declaration amount field, and corridor exclusion logic ensuring tax calculations only apply to transactions routed through the specific partner — not all transactions for that currency.

12 Learnings
Lessons & Insights
● What Worked Well

Grace Period Strategy

Introducing new API parameters as non-mandatory initially gave clients a transition window without breaking existing integrations. This approach enabled compliance enforcement on a timeline rather than creating a hard cutoff that could have disrupted active payment flows for dozens of API clients.

● What Worked Well

Proactive Client Communication

Drafting and sending a structured notification email before the changes went live gave API clients advance notice of the new parameters. This prevented support escalations and demonstrated professional change management to the partner ecosystem.

● Challenge

Partner Rejection Required Pivot

The initial proposal to calculate tax internally was rejected by the partner, requiring a complete pivot to their API integration approach. This added complexity (sequential API calls, failure handling, timeout scenarios) that would not have existed with internal calculation. Early alignment on integration approach would have saved analysis time.

● Key Learning

Corridor Exclusion Is Critical

Post-release discovery that tax was being applied to non-partner payment links for the same currency highlighted the importance of routing-level exclusion logic. The fix was shipped within days, but the edge case should have been identified during specification review rather than in production.

● Risk Identified

Partner API Reliability Dependency

The platform's tax accuracy is now dependent on the partner's API availability. While fallback messaging handles outages gracefully, extended downtime could impact payer experience at scale. The tcs_api_fail database flag enables monitoring, but a long-term internal calculation fallback may be needed.

● Reflection

What I'd Do Differently

Would have pushed harder for a hybrid approach during initial partner negotiation — using their API as primary but maintaining an internal calculation as fallback. The current architecture has a single point of failure on the partner's API. A cached rate approach with periodic refresh could provide resilience without violating the partner's requirement for API-based integration.
