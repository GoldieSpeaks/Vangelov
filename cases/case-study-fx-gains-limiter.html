Financial Logic
FX Conversion Gains Limiter
I designed and deployed a financial capping mechanism to limit FX conversion revenue on payment transactions — ensuring compliance with a national government tender requirement across 8 institutional payment links.

I identified a mandatory clause within a national government tender requiring the platform to place a hard cap on foreign exchange conversion gains for specific institutions. I designed, validated, and deployed the capping logic across both API and web payment systems within a compressed timeline to meet tender compliance deadlines, securing written approval from the COO and Commercial Director before release.

5 Overview
Executive Summary

I implemented a financial capping mechanism to limit foreign exchange conversion gains on student payment transactions. I identified the requirement from a government tender clause that mandated maximum FX revenue thresholds for specific institutions participating in the tender framework.

I designed the calculation model, validated it with senior stakeholders (COO and Commercial Director), coordinated implementation across both API and web payment systems, and managed deployment to 8 institutional payment links. I ensured the logic correctly excluded same-currency transactions and handled third-party payment provider scenarios.

I completed the implementation, coordinated testing across all environments (internal and production), and secured final written approval. The tender compliance requirement was met, enabling the platform to retain and secure institutional contracts.

12 Learnings
Lessons & Insights
● What Worked Well

Spreadsheet-Based Logic Validation

Using a calculation spreadsheet (CAP Calc) to validate the FX capping logic before development allowed stakeholders to verify the formula and edge cases in a format they could understand. This eliminated ambiguity and accelerated the sign-off process.

● What Worked Well

Staged Deployment Strategy

Rolling out to a demo link first, then all test links, then production ensured each stage was validated before escalating. The 2-day gap between production deployment and external testing provided a safe buffer for monitoring.

● Challenge

Infrastructure Dependency

The project was blocked by an ongoing PHP upgrade. Initial test environment changes were lost, requiring re-implementation. This added approximately one month to the timeline and highlighted the need to coordinate with infrastructure work.

● Key Learning

Financial Logic Requires Formal Trails

For any change affecting revenue calculations, formal written sign-off is essential rather than verbal approval. The requirement to wait for a specific stakeholder to return from leave reinforced that financial logic changes need documented approval chains.

● Risk Identified

Hardcoded Values Limit Flexibility

The decision to hardcode the cap threshold was pragmatic for speed but creates maintenance overhead if the tender terms change. A future iteration should consider a configurable threshold that can be adjusted without code deployment.

● Reflection

What I'd Do Differently

Would have identified the PHP upgrade dependency earlier and aligned the project kickoff with the infrastructure timeline. The lost test environment work was preventable with better cross-team visibility into the upgrade schedule.
