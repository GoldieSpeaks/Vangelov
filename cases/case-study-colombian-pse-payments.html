Payment Integration
Colombian PSE Payment Integration
I led the integration of Colombia's PSE electronic bank transfer method across three payment channels — replacing an existing provider, enabling invoice-based bulk payments, and opening new revenue corridors in the Latin American market.

I identified that the cross-border payments platform needed to enable bank transfer payments from Colombia using PSE (Pagos Seguros en Línea), a widely adopted regional electronic payment method supported by over 20 banks. The existing provider had critical limitations — it could not support invoice-based bulk payments where individual payer details are not collected. I coordinated the integration of a new regional payment provider using a redirect-based flow across three payment channels, bundling a parallel API bank transfer separation to future-proof the architecture.

5 Overview
Executive Summary

I integrated PSE (Pagos Seguros en Línea) — Colombia's most widely adopted electronic bank transfer method — across three payment channels through a new regional payment provider. The existing provider could not support invoice-based bulk payments and was more expensive, driving my strategic decision to replace it entirely.

I coordinated with external partners for sandbox credentials, settlement model alignment, and production validation with penny tests. I bundled a parallel API refactoring effort to separate bank transfer logic and accommodate the new payment method architecture. The resulting integration successfully replaced the previous provider across all channels and opened Colombian bank transfer payments to the invoice API for the first time, enabling bulk payments without individual payer details.

8 Scope
Project Scope

In Scope

  • Web payment links — Electronic bank transfer via redirect
  • Payer API — Redirect URL in payment response
  • Invoice API — Bulk payments with NIT (Tax ID) parameter
  • Operations — COP recalculations on active orders
  • Colombian GMF tax (0.4%) added to payment amounts
  • Bank transfer separation (parallel API refactor)
  • Settlement model alignment (FX to USD operational account)

Out of Scope

  • Partner education system payments
  • Agent payment links
  • Legacy API users (no redirect support)
  • Rate of exchange display on user interface

12 Learnings
Lessons & Insights
● What Worked Well

Provider Redirect Simplification

The provider's decision to handle bank details collection on their redirect page significantly reduced integration complexity. The platform only needed to pass core payment parameters and handle the redirect flow, eliminating complex bank selection UI requirements.

● What Worked Well

Bundled Bank Transfer Separation

Combining the bank transfer separation refactor with this project ensured the API architecture was future-proof. This parallel effort enabled cleaner integration for subsequent regional payment corridors without requiring additional refactoring.

● Challenge

External Dependency on Credentials

Waiting on production credentials from the regional provider introduced delays. Sandbox testing progressed but production validation could only happen after credentials were provided, creating a bottleneck in the final weeks.

● Challenge

Settlement Model Cross-Team Alignment

Determining the settlement currency and account required coordination between treasury, the commercial team, and the provider. This cross-team alignment took several weeks to finalize and blocked BRD completion.

● Key Learning

Regional Tax Complexity

Colombia's 0.4% GMF tax requirement was discovered during scoping and had to be factored into the payment calculation. Regional integrations must always account for local tax requirements that may not be obvious from API documentation alone.

● Reflection

What I'd Do Differently

Would have initiated the settlement model alignment and production credential requests in week one rather than waiting for the scoping call outcome. The sequential dependency on external partners for both credentials and settlement decisions added avoidable weeks to the timeline.
