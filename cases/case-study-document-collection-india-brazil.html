Security & Compliance
Cross-Border Document Collection Pipeline
I designed and delivered a mandatory document collection system for two regulated payment corridors — replacing an unreliable email-based process with an automated upload pipeline connected to regional cloud infrastructure via secure file transfer.

I identified that regulatory requirements in two high-volume corridors demanded that payers submit identity and compliance documents before payments could be processed. The existing manual process relied on payers emailing documents to a partner address — a method frequently neglected, causing expired transactions, delayed processing, and significant operational overhead for the regional partner. I built and coordinated an end-to-end upload pipeline with automated packaging, secure transfer to regional cloud storage, and transaction expiration logic.

5 Overview
Executive Summary

I delivered a mandatory document collection system for two regulated payment corridors where compliance documentation is required before funds can be processed. The prior process — asking payers to email documents to a partner mailbox — was unreliable, frequently ignored, and created significant operational burden on the regional partner.

I designed an integrated upload interface shown immediately after payment registration, with country-specific document slot requirements, automated ZIP archiving, secure file transfer to a regional cloud server via SFTP, and a 24-hour expiration window after which incomplete transactions are automatically invalidated. I specified partial upload support with resumption, progress indicators, and clear user guidance.

All 21 QA test cases passed across three internal environments. I released the feature to production and it is processing transactions across both corridors with zero documents lost in transit since launch.

7 Delivery
Project Timeline

End-to-end delivery from requirement identification through development, partner validation, and production release.

Week 1
Requirement Identification & Epic Creation
I identified the regulatory requirement for document collection in two corridors and created the project epic. I held the initial scoping call with business stakeholders to define the document types required per country and payer category.
Week 2
SFTP Infrastructure Request
I raised the request for a new secure file transfer instance located in the regulated region to comply with data residency requirements. IT teams confirmed feasibility and began configuration of the regional cloud storage.
Week 3
BRD & Analysis
Business Analyst authored the functional specification covering document slot logic, payer-type differentiation, expiration rules, ZIP packaging format, and SFTP naming conventions. I reviewed and approved the specification.
Week 4
Sign-Off & Key Decisions
I held the sign-off call with stakeholders and made the decision to set 24-hour expiration for incomplete transactions. I assigned the developer and moved the project to active development, and allocated the QA engineer.
Weeks 5-6
Core Upload System Development
New upload system built with country-specific document slots, mandatory field validation, progress indicator, and automated ZIP archiving. Database schema extended with new tables for upload groups, fields, conditions, and ZIP tracking.
Week 6
Partner Validation & Prototypes
I sent prototypes with upload field labels and instructions to the regional partner for approval. Partner confirmed document requirements and validated that the upload field structure met their processing needs.
Week 7
Payment Instructions Rework
Payment instructions for both corridors updated to reflect the new document upload process. Old email-based instructions removed. New hyperlinks added to guide payers to required document information before upload.
Weeks 7-8
QA Testing Across Environments
QA engineer tested all scenarios across three internal environments covering different currency configurations. All 21 test cases passed including upload flows, expiration logic, SFTP delivery, and partial upload resumption.
Weeks 8-12
Release Rescheduling
Release moved through three delivery windows (August, September Part 2, September Part 3) due to competing priorities across the platform. I actively managed version reassignment to maintain feature visibility.
Week 13

I released the feature to production in the October release window. Post-launch, I added a hyperlink enhancement to improve pre-upload information display. System processing live transactions across both corridors with zero document loss.

12 Learnings
Lessons & Insights
● What Worked Well

Partner Validation Before QA

Sending UI prototypes to the regional partner for approval before formal QA testing eliminated rework cycles. The partner confirmed document slot labels and requirements upfront, meaning the QA team tested the final version first time rather than iterating through feedback rounds.

● What Worked Well

Critical Path Focus

Deferring the informational popup to a separate developer kept the primary release on track. The popup was a nice-to-have enhancement, not a blocking requirement. This pattern of separating enhancements from core functionality maintained delivery velocity.

● Challenge

Three Release Reschedulings

The feature was ready in August but didn't reach production until October due to competing platform priorities. Each rescheduling required version reassignment and stakeholder communication. Active release management prevented the feature from being deprioritised entirely.

● Key Learning

Expiration Window Requires Balancing

The initial development used a 72-hour window, which was then reduced to 24 hours during sign-off. This taught the importance of deciding business rules early in the process rather than during development — the timeframe change required code adjustments that could have been avoided with earlier alignment.

● Risk Identified

CRM Automation Gap

The CRM-side cron job for automatic transaction cancellation was deferred post-release due to developer availability. This created a temporary gap where expired transactions required manual intervention until the automation was implemented in a subsequent release.

● Reflection

What I'd Do Differently

Would establish a "release buffer" policy — booking a release window one iteration ahead of readiness to absorb rescheduling. Three consecutive moves from August to October could have been reduced to one if the initial target had been set more conservatively given known competing priorities.
