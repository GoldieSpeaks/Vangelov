Payment Integration
Egypt Fawry Payment Integration
I delivered the local currency bank transfer integration for Egyptian payers via the Fawry network, unlocking a new market corridor by extending an existing provider with minimal custom development.

I identified that Egyptian payers were limited to foreign-currency payments (EUR, GBP, AUD) via bank transfer, creating friction and reducing conversion rates. I leveraged an existing provider integration to add a local payment method supporting Egyptian Pounds (EGP), coordinating across 29 emails with the provider, commercial, and development teams to deliver in 7 weeks without new architecture.

5 Overview
Executive Summary

I extended an existing third-party payment provider integration to enable local currency bank transfers for Egyptian payers via the Fawry payment network. Since the provider integration was already in place, I determined no formal BRD was required — this was a configuration and enablement exercise rather than a greenfield build.

I deployed the integration to production test links for penny testing before full enablement. Two post-release defects were identified and I coordinated their resolution: an incorrect API redirect link and a missing transaction details button. Following successful launch, I expanded the scope to additional payment link types after obtaining senior stakeholder approval.

Payers in Egypt can now complete local currency bank transfers through a familiar, nationally-supported payment method — replacing what was previously a foreign-currency-only barrier to conversion.

12 Learnings
Lessons & Insights
● What Worked Well

Leveraging Existing Integration

Because the provider integration already existed, enabling a new country and payment method required configuration rather than new architecture. This reduced delivery time to 7 weeks and eliminated BRD overhead entirely.

● What Worked Well

Proactive Coordination

With 29 emails across the project lifecycle, active stakeholder coordination ensured sandbox issues were escalated promptly, margin rates were corrected before launch, and post-launch expansion received timely approval.

● Challenge

Provider Sandbox Reliability

The provider's sandbox environment had issues requiring escalation and multiple follow-ups. Third-party test environments cannot be assumed production-ready. Building buffer time for provider-side issues proved essential.

● Key Learning

Exception Documentation Is Critical

The payment method had multiple platform exceptions (partners, invoice API, old API versions, specific sponsor flows). Documenting these clearly prevented false-positive bug reports and set correct expectations with QA and operations teams.

● Risk Identified

Downstream Data Sync Gaps

A separate defect emerged where transactions using the new payment method were not syncing correctly to the corporate operations system. New payment methods need end-to-end validation including all downstream data flows, not just the payer-facing checkout.

● Reflection

What I'd Do Differently

Would have mapped all payment link types from the start rather than discovering gaps post-launch. The follow-up effort to enable Fawry for additional link categories took weeks of coordination and multiple calls. A comprehensive link-type audit upfront would have allowed a single, complete rollout.
