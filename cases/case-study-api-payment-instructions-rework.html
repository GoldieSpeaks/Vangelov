API Payment Instructions Rework
I designed and documented the replacement of legacy HTML payment instructions with a structured XML data model and TM-hosted payment page — eliminating broken rendering across all API partner surfaces and giving clients parseable, secure payment guidance across 5 regions and 14 languages.

The Student Payments API returned bank transfer instructions as pre-rendered HTML blobs with CSS class references to an unavailable internal stylesheet. Students saw broken formatting at the most critical payment moment. I proposed and fully documented a layered solution: structured XML for full client control, plus a branded hosted page for zero-effort integration — covering all regional variants with zero breaking changes.

5 Overview
Executive Summary

I identified that the education payments API was returning bank transfer instructions as pre-rendered HTML with CSS class references to an internal stylesheet clients never had access to. Students saw broken formatting at the most critical moment — handling high-value international tuition transfers.

I proposed and fully documented a layered solution: (1) a new structured XML element providing typed, parseable data that clients can render with their own design system, and (2) a TM-hosted payment instructions page for zero-effort integration. The solution covers 5 regions, 14 languages, and preserves full backward compatibility with zero breaking changes.

I produced the BRD, XSD schema, 15 sample payloads, hosted page architecture, wireframes for all regions, interaction specifications, and two HTML prototype mockups. The initial alignment phase went live in a prior release; the structured data rework is in active development.

9 Architecture
Technical Architecture

Layered solution providing two integration paths for API clients.

10 Decisions
Key Decisions
When Decision Type
Phase 1 Mirror web platform instructions on API. Stakeholder consensus that API should use identical instruction text to the web platform as an immediate alignment step. Strategic
Phase 2 Additive structured XML element — no new API version. New element appears as a sibling; clients detect via presence. Existing parsers see no change whatsoever. Technical
Phase 2 Namespace-versioned schema (v1). Enables future breaking changes without affecting existing parsers. Follows XML best practice for extensibility. Technical
Phase 2 JWT token contains NO bank details. Only transaction ID reference stored in token; financial data fetched server-side. Eliminates data exposure risk. Technical
Phase 2 12-month minimum deprecation period. Both HTML blob and structured XML present simultaneously during transition. Gives clients adequate migration runway. Process
Phase 2 Two migration paths offered. Clients choose: parse structured XML (medium effort, full control) or redirect to hosted page (low effort, no control). Accommodates different client capabilities. Strategic

12 Learnings
Lessons & Insights
● What Worked Well

Layered Solution Architecture

Offering two integration paths (structured XML for full control vs. hosted page for zero effort) accommodates the full spectrum of client technical capability — from sophisticated platforms with custom renderers to small colleges that just need a redirect URL.

● What Worked Well

Backward Compatibility Preserved

By making the structured XML purely additive and running both formats in parallel for 12 months, the project eliminates integration risk for existing clients while enabling a clear migration path forward.

● Challenge

Infrastructure Pause Caused Re-Analysis

A platform-wide infrastructure migration paused Phase 1 for several months. When work resumed, a full re-analysis was required because initial context had been lost. Documentation became critical for continuity.

● Key Learning

Regional Complexity Is a Product in Itself

India requires 6-step processes, 11 documents, TCS tax, and relative eligibility rules. China needs dual bank accounts and full bilingual content. Each regional variant is essentially a separate product requiring dedicated analysis.

● Risk Identified

Data-Presentation Coupling in Legacy APIs

The original API was designed when all instructions rendered inside the internal web application. When extended to serve external clients, the HTML was reused without removing the stylesheet dependency — a pattern that created years of poor client experience.

● Reflection

What I Would Do Differently

Would have pushed for the structured data approach from the start rather than the intermediate alignment step. While the phased approach was politically necessary for stakeholder buy-in, the technical debt of maintaining the HTML blob format adds ongoing complexity that could have been avoided with a more ambitious initial proposal.
