5 Overview
Executive Summary

I delivered the Minimum Viable Product of a Payment Analytics Dashboard for API-integrated education payment clients. Prior to this initiative, these clients relied exclusively on programmatic API access for payment data — no visual analytics, no reporting tools, no self-service interface existed. Competitors already offered equivalent capability, making this a direct source of client churn.

I drove the project end-to-end: creating wireframes and low-fidelity UX designs (login, account creation, dashboard pages), co-authoring the BRD, defining account management logic, coordinating pilot client data validation, and driving the ISO 27001 penetration testing process. I released the MVP to pilot clients within 14 months, with only 1 informational finding from the security assessment.

Post-MVP client feedback drove a full design overhaul that I presented to three senior executives, receiving enthusiastic C-level approval. The project established the foundation for Phase 2 features including disbursements, white-labelling, and accommodation provider integration.

6 Delivery
Project Timeline

End-to-end delivery from demand request through MVP release, client pilot, and design overhaul approval.

Phase 1 — Initiation
Demand Request & Scoping
Business leadership submitted demand request citing competitive gap. I conducted initial scoping calls, outlined page structure (Home, Payments, Documentation, Help, API Docs), and established the project framework.
Phase 2 — Analysis
BRD Co-Authoring & Sign-Off
I co-authored the functional specification with the Business Analyst. Defined account logic (master/sub hierarchy), transaction table attributes, refund handling, and data transfer requirements. BRD signed off by business stakeholders.
Phase 3 — Design
Wireframing & Page Architecture
I created all wireframes establishing visual direction and navigation. Defined analytics layout, payment order views, account creation flows, and reporting module structure. I assembled presentation material with payment statistics examples for stakeholder alignment.
Phase 4 — UX Design
Wireframing & Low-Fidelity Design
I created low-fidelity wireframes and UX designs for all key pages: login, account creation, password recovery, main dashboard layout, and user management section. Delivered as design packages for the development team to implement.
Phase 5 — Development
Back-End Integration & Data Layer
Lead developer built the front-end from my wireframe specifications and integrated with back-end logic. I oversaw database structure creation for payment data, account management, and analytics. Dedicated subdomain configured as the access point. I prepared the pilot client list for data validation.
Phase 6 — QA
Internal Testing & QA Validation
I coordinated QA through internal and external test cycles across payment flows, account management, and reporting modules. Reviewed terms and conditions with legal. All functional tests passed without critical issues.
Phase 7 — Security
ISO 27001 Penetration Testing
I identified the compliance requirement, sourced an external security vendor, managed SOW approval with senior stakeholders, and coordinated a dedicated test environment. Assessment returned only 1 informational finding — no critical vulnerabilities.
Phase 8 — Release
MVP Launch & Pilot
I managed the release via SysAid release ticket. Created master accounts and distributed to pilot clients. I initiated feedback collection immediately post-launch, establishing the data-driven improvement cycle.
Phase 9 — Feedback
Client Feedback & Iteration
Pilot clients provided structured feedback: status descriptions needed, student ID filtering requested, sortable columns required, and external reference ID display prioritised. I synthesised findings into an improvement roadmap.
Phase 10 — Overhaul
Design Overhaul & C-Level Approval
Based on client feedback, I proposed a comprehensive design overhaul. Presented to COO, International Commercial Director, and CPO. All three approved the new direction, describing it as “so much cleaner.” Foundation set for Phase 2 features.

11 Learnings
Lessons & Insights
● What Worked Well

Owning UX Design Directly

Having the Product Manager create wireframes and low-fidelity designs directly eliminated the typical handoff delay between product vision and development. The development team received clear, detailed design specifications that reduced ambiguity, cutting weeks from the schedule and enabling rapid iteration on stakeholder feedback.

● What Worked Well

Security-First Release Gate

Making penetration testing a mandatory gate before any client release validated the security posture early. The single informational finding gave stakeholders confidence and eliminated post-release security concerns that could have delayed adoption.

● Challenge

Data Model Alignment Complexity

The mismatch between back-office client file IDs and API client structures created integration complexity. Not all API clients had corresponding back-office file references, requiring careful mapping logic and additional alignment calls to resolve.

● Key Learning

MVP Speed vs Design Quality Trade-off

Releasing a functional-but-unpolished MVP generated more accurate client signals than upfront design investment would have. The real client feedback that drove the overhaul was more valuable than any assumptions made during the design phase.

● Risk Identified

Scope Expansion Post-MVP

Post-release success generated rapid scope expansion requests: white-labelling, disbursements, accommodation providers, year-over-year statistics. Without active scope management, each request could have delayed the design overhaul that was the highest-priority improvement.

● Reflection

What I'd Do Differently

Would have engaged a dedicated UX resource from the start to elevate the low-fidelity wireframes into high-fidelity designs. While the PM-driven UX approach accelerated delivery, the gap between “functional wireframes” and “polished production design” was wider than anticipated. The post-MVP design overhaul cost more effort than early UX investment would have required.
