5 Overview
Executive Summary

I identified that the cross-border payments platform used a timer embedded in its back-office operations application to automatically cancel payment lines that remained in "Setup" status for 14 or more days. This timer fired every 4 hours from every running instance of the application, creating significant redundant database traffic and introducing race conditions where multiple instances would attempt to process the same records simultaneously.

I defined and delivered a migration of the entire cancellation mechanism to a database-level CRON job. I specified the materialized view approach to efficiently identify qualifying records, and a scheduled function that runs hourly to perform the status transitions. The migration eliminated all application-level traffic for this process, resolved the multi-instance race condition, and increased detection frequency from every 4 hours to every hour.

I delivered in approximately 4 weeks from initial request to production validation, with all test cases passing on the first attempt. Production verification confirmed correct behaviour across real transaction data spanning multiple operational regions.

12 Learnings
Lessons & Insights
● What Worked Well

Database-Level Architecture

Moving the logic to the database layer eliminated an entire class of problems — race conditions, redundant traffic, and multi-instance synchronization complexity. The simplicity of a single CRON job proved more maintainable and more performant than any application-level locking solution would have been.

● What Worked Well

Materialized View Approach

Using a pre-computed materialized view rather than running expensive queries on each execution provided a clean separation between data identification and action execution. This pattern proved reusable for other scheduled database operations.

● Key Learning

Production Validation is Essential

The diversity of real transaction data — varying statuses, timestamps spanning edge cases, and cross-regional patterns — made production testing the only way to achieve full confidence. Test environments with synthetic data cannot replicate this complexity at scale.

● Challenge

Legacy Timer Patterns at Scale

The original timer pattern had been in place long enough that its inefficiency was normalized. The performance impact only became clearly visible when analyzing aggregate database traffic patterns. Proactive monitoring of shared-resource access patterns helps surface similar inefficiencies earlier.

● Risk Identified

Multi-Instance Processes Need Coordination

Any process that runs independently across multiple application instances risks duplication and race conditions unless explicitly coordinated. This pattern likely exists in other parts of the system. The CRON migration proved that centralizing such processes at the database level is the cleanest solution.

● Reflection

What I'd Do Differently

Would have requested a broader audit of all timer-based processes across the application suite alongside this improvement. The same distributed timer pattern likely affects other operations. Addressing them as a batch would have delivered compounding efficiency gains and established a standard pattern for future scheduled tasks.
