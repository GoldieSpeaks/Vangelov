Payment Integration
Cross-Border Payment Network Integration
I delivered a cross-border payment network integration that opened a new South-East Asian corridor — enabling local-currency bank transfers for students in a previously unreachable market within 8 weeks.

I identified a global payment network provider as the optimal routing partner, scoped the integration end-to-end, and coordinated delivery of API-based virtual account infrastructure that enabled real-time KRW payment instructions for South Korean students. I took the project from initial vendor evaluation to live deployment in under 8 weeks, and subsequently managed the provider's controlled offboarding when the corridor was absorbed by an alternative route.

5 Overview
Executive Summary

I integrated a global cross-border payment network provider as a third-party payment facilitator, enabling local-currency (KRW) bank transfer payments for students in South Korea. The platform previously had no mechanism to serve this corridor, creating a gap in market coverage that competitors already addressed.

I designed the integration to leverage the provider's API for creating payment orders, receiving dynamically-assigned virtual account numbers, and processing webhook confirmations upon successful fund receipt. I coordinated the solution's deployment across both API-integrated clients and standalone student payment portals, with funds settled in USD before distribution to educational institutions.

I subsequently managed the corridor's migration to an alternative payment route and formally offboarded the provider. This full lifecycle — from integration through operational use to controlled decommissioning — demonstrated my vendor management capabilities and commitment to clean architectural separation.

7 Delivery
Project Timeline

Rapid integration cycle from demand through to live deployment and eventual controlled offboarding.

Week 1
Demand Request & Vendor Identification
Business leadership identified the need for South Korean corridor coverage. I evaluated potential providers and selected the cross-border payment network based on their virtual account infrastructure and API capabilities.
Week 1-2
Initial Scoping & BRD Creation
I completed initial scoping and authored the functional specification covering payment flow, API integration points, static/dynamic bank account configuration, and settlement routing logic.
Week 2
API Documentation & Field Mapping
I shared the Postman collection with the development team. I documented key field mappings: external_id (platform unique ID), company_name (school ID), bank_id (optional, default SHINHAN BANK 088). I clarified static vs. dynamic account details.
Week 2-3
Development Begins
Lead developer started implementation for API clients and standalone student portals. Integration architecture established with payment order creation, virtual account assignment, and webhook handling.
Week 3
Provider API Issue & Resolution
Technical error encountered in the provider's API returning generic error responses. Implementation paused. I escalated directly to the provider's technical contact and resolved the issue within 24 hours.
Week 3
Implementation Complete
Payment process fully implemented for API clients and standalone student portals. I documented limitations: Invoice API, Partner system, and Agent flows excluded due to payment process incompatibility.
Week 4
User Flow Documentation & QA
I created end-to-end user flow diagrams. I assigned the QA engineer for testing and defined test cases covering both internal validation and external production verification.
Week 5
Provider Sandbox & Refunds Setup
Provider delivered sandbox access to their refunds dashboard. I reviewed the Business Hub interface for operational readiness and refund processing capabilities.
Week 6-8
Release & Go-Live
Integration released as part of the scheduled web deployment cycle. South Korean corridor live for all eligible payment flows. Payment instructions displaying dynamic virtual account numbers to students.
Post-Launch
Controlled Offboarding
Corridor subsequently migrated to alternative payment route. I managed the full offboarding process: confirmed system removal with the development team, coordinated with treasury and financial services, and ensured clean decommissioning across all platform components.

12 Learnings
Lessons & Insights
● What Worked Well

Lean Documentation for Clear-Scope Projects

Bypassing a full BRD in favour of a focused specification I authored, with direct API documentation sharing, saved weeks of analysis overhead. When the scope is narrow and the API contract is well-defined, lightweight documentation accelerates delivery without sacrificing clarity.

● What Worked Well

Direct Provider Contact for Blockers

Escalating the API error directly to the provider's named technical contact rather than going through support channels resolved a potential multi-day blocker within 24 hours. Having a named contact established during onboarding proved invaluable.

● Challenge

Provider Dependency Risk

The integration created a single-point dependency for the South Korean corridor. When the provider was subsequently offboarded, the corridor needed migration to an alternative route. This highlighted the importance of corridor redundancy planning for critical markets.

● Key Learning

Scope Exclusions Require Documentation

The incompatibility with Invoice API, Partners, and Agent flows was discovered during development rather than scoping. Earlier technical feasibility checks across all platform channels would have set expectations sooner and avoided mid-sprint scope questions.

● Risk Identified

Vendor Lifecycle Management

The full lifecycle from onboarding to offboarding reinforced that integration projects need decommissioning plans from day one. Without clean architectural separation, removing a provider from multiple system touchpoints can become a complex operation.

● Reflection

What I'd Do Differently

Would establish contractual exit clauses and technical decommissioning procedures during the integration phase rather than defining them reactively at offboarding time. A "vendor exit playbook" created at onboarding would have streamlined the eventual offboarding process.
